실습 개요

배포된 프로덕션 모델에 고객의 웹 애플리케이션을 통합하는 과제를 맡게 되었습니다. 현재 웹 애플리케이션을 확인하고, 배포된 엔드포인트에 액세스하는 방법을 조사하고, 고객의 웹 애플리케이션에 통합하여 추론을 위한 완전한 솔루션을 전달하는 방법을 선택합니다. 고객은 현재 정부 프로그램의 요구를 기준으로 하는 지원 요청에 대한 모든 세부 정보가 포함된 양식의 웹 페이지를 보유하고 있습니다. 고객은 이전 실습에서 생성한 모델과 엔드포인트를 이 웹 사이트에 통합하려고 합니다. 양식이 채워지면 모델은 후보를 평가하고, 요구를 기준으로 하는 지원에 적합한지 여부를 출력합니다.

이 실습에서는 웹 애플리케이션을 확인합니다. Amazon Elastic Compute Cloud(Amazon EC2) 인스턴스를 사용해서 Amazon API Gateway API를 HTML 웹 페이지에 추가하여 Amazon Simple Storage Service(Amazon S3)에 호스팅된 웹 애플리케이션을 업데이트합니다. 그런 후 새 데이터로 실시간 엔드포인트를 테스트합니다.

실습 환경

이 실습에서 웹 애플리케이션 사용자는 Amazon S3에서 정적 HTML 페이지에 액세스하고 POST 메서드를 사용해서 양식 데이터를 API Gateway로 전송합니다. POST 메서드는 데이터를 Amazon SageMaker 엔드포인트가 해석할 수 있는 형식으로 변환하는 Lambda 함수를 시작합니다. Amazon SageMaker 엔드포인트는 이 입력을 수락하고, 미리 훈련시킨 모델을 통해 실행하고, AWS Lambda로 반환되는 새 예측을 생성합니다. AWS Lambda는 API Gateway로 결과를 반환하여 사용자에게 예측을 보냅니다.

아래 다이어그램에 실습 환경의 기본 아키텍처가 나와 있습니다.

이미지 설명: 이전 다이어그램은 Amazon SageMaker 모델이 있는 Amazon Simple Storage Service(Amazon S3) 버킷을 보여 줍니다. 추론 코드가 있는 Jupyter Notebook은 SageMaker 컨테이너 이미지의 Amazon ECR 리포지토리에 있는 추론 컨테이너 이미지를 사용합니다. 이 노트북 코드는 SageMaker 엔드포인트를 사용합니다. 웹 애플리케이션은 Amazon S3 버킷에 호스팅되고 Amazon API Gateway 및 AWS Lambda 함수를 사용해서 SageMaker 엔드포인트에 대한 추론을 수행합니다. 웹 애플리케이션 사용자는 웹 애플리케이션에 액세스하는 것으로 표시됩니다.

과제 1: Amazon S3 및 Amazon API Gateway의 웹 애플리케이션을 검토하고 테스트

고객의 웹 애플리케이션은 서버리스 애플리케이션을 작성하기 위한 프레임워크인 Chalice를 사용하여 이미 생성되었습니다. 이 애플리케이션에서는 이전 실습에서 생성한 SageMaker 엔드포인트가 웹 사이트에서 POST API를 호출할 때 API Gateway에 의해 호출된 Lambda 함수와 통합되었습니다. 현재 index.html 페이지에는 API Gateway API에 대한 링크가 포함되어 있지 않습니다. 웹 사이트 리소스를 확인하고, 요구를 기준으로 하는 지원 요청이 제출될 때 웹 사이트가 현재 반환하는 출력을 평가합니다.

이 과제에서는 웹 애플리케이션을 검토하고 테스트합니다. Amazon S3에 호스트된 정적 웹 사이트를 확인한 다음, Amazon API Gateway API를 확인합니다.

먼저, Amazon S3 버킷에서 웹 사이트 파일을 확인합니다.

  1. AWS 관리 콘솔 상단의 검색 창에서

    Copied
    를 검색하여 선택합니다.

  2. 버킷 목록에서 이 지침 왼쪽에 나열된 S3BucketName 값과 일치하는 Amazon S3 버킷을 엽니다.

참고: Objects 아래에 다음이 표시됩니다.

  • index.html – 웹 사이트 호스팅을 사용하도록 설정하려면 인덱스 문서를 구성하고 업로드해야 합니다. 인덱스 문서는 웹 사이트의 루트 또는 하위 폴더에 대해 요청이 수행될 때 Amazon S3가 반환하는 웹 페이지입니다. 예를 들어, 사용자가 브라우저에서 http://www.example.com으로 이동하면 특정 페이지를 요청하지 않게 됩니다. 이 경우 Amazon S3는 기본 페이지로도 참조되는 인덱스 문서를 제공합니다.

  • error.html – 버킷을 정적 웹 사이트로 구성한 후에 오류가 발생하면 Amazon S3는 HTML 오류 문서를 반환합니다. 필요에 따라 오류가 발생할 때 Amazon S3가 해당 문서를 반환하도록 사용자 지정 오류 문서로 버킷을 구성할 수 있습니다. 이 실습에서는 HTTP 404 - Page not found error에 대한 오류 문서가 있습니다.

  • income-model.tgz – 이것은 API Gateway에 연결된 애플리케이션 tar 파일입니다. 이 애플리케이션 tar 파일의 내용은 이 실습의 일부로 이미 프로비저닝된 app.py 파일에 저장되어 있습니다. 사용자가 프런트엔드 애플리케이션을 사용해서 요청을 제출하면 API Gateway로 이동됩니다. 이렇게 하면 Chalice 애플리케이션에 포함된 app.py 파일을 기준으로 실행되고 SageMaker 엔드포인트로 데이터를 전송하는 Lambda 함수가 트리거됩니다. 필요한 모든 전처리는 사용자 지정 app.py 파일에서 수행할 수 있습니다. 다음은 이 실습에 대해 프로비저닝된 app.py 파일의 내용입니다. 잠시 시간을 내서 app.py 파일의 내용을 확인하십시오.

@app.route('/', methods=['POST'], content_types=['application/x-www-form-urlencoded'])
def handle_data():
    d = parse_qs(app.current_request.raw_body)
    # data to csv

    try:
        my_dict = {k:float(v[0]) for k, v in d.iteritems()}
        print(my_dict)
    except AttributeError:
        my_dict = {k:float(v[0]) for k, v in d.items()}
        print("There was an error")
        print(my_dict)
    f = StringIO()
    w = csv.DictWriter(f, my_dict.keys())

    w.writerow(my_dict)
    print("Ready to send request:")
    print(f.getvalue())
    res = sagemaker.invoke_endpoint(

                    Body=f.getvalue(),

    print("Request complete")
    print(res)
    res_float = float(res['Body'].read())
    if res_float < 0.5:
        evaluation = f'This candidate is not eligible for government assistance.'
    else:
        evaluation = f'This candidate is eligible for government assistance.'
    return json.dumps(evaluation)

과제 2: 애플리케이션을 SageMaker 엔드포인트에 연결

웹 애플리케이션 및 SageMaker 엔드포인트를 함께 연결하려면 index.html 파일에 API Gateway API 링크를 추가해야 합니다. EC2 Instance Connect가 있는 EC2 인스턴스가 실습 설정의 일부로 생성되었습니다. 이 EC2 인스턴스를 사용해서 Amazon S3에서 index.html 파일을 다운로드하고, API Gateway API로 파일을 업데이트한 다음, 해당 파일을 S3 버킷으로 업로드합니다.

이 과제에서는 EC2 인스턴스를 사용해서 index.html 파일을 편집하고 변경 사항을 S3 버킷에 업로드합니다.

  1. AWS 관리 콘솔 상단의 검색 창에서

    Copied
    를 검색하여 선택합니다.

  2. 왼쪽 탐색 창의 Instances에서 Instances를 선택합니다.

  3. CommandHostLinux 인스턴스를 선택합니다.

  4. 탐색 모음에서 Connect를 선택합니다.

  5. Connect to instance 창에서 EC2 Instance Connect 탭을 선택하고 Connect를 선택합니다.

터미널 세션은 새 브라우저 탭에서 열립니다.

과제 3: 애플리케이션에서 새 데이터 레코드에 대한 추론 가져오기

이제 웹 애플리케이션이 API Gateway에 연결되었으므로 웹 사이트를 테스트하고 추론 결과를 다시 가져올 수 있습니다. 웹 사이트를 열고 새 데이터 레코드에 대한 추론을 가져옵니다.

이 과제에서는 웹 애플리케이션을 테스트합니다.

  1. 이 지침의 왼쪽에 나열된 WebsiteURL 값을 복사합니다. 새 웹 브라우저 탭에 URL을 붙여넣고 Enter 키를 눌러 애플리케이션 웹 사이트로 이동합니다.

  2. 각 범주 아래에서 빈 상자에 해당 값을 제공합니다.

참고: 추론 해석에 도움을 주기 위해 범주 및 해당 값의 키가 다음 목록에 포함되어 있습니다.

  • Age: 0~99(년)
  • Work class: 0~3(각각 개인, 정부, 자영업 및 기타)
  • Education: 0~5(각각 고등학교 졸업, 전문대학, 학사, 중퇴, 석사 및 박사)
  • Education number: 0~15(각각 9, 10, 13, 14, 11, 7, 12, 6, 4, 15, 5, 16, 8, 3, 2, 1년)
  • Marital status: 0~3(각각 기혼, 미혼, 이혼, 사별)
  • Occupation: 0~4(각각 노동자, 서비스, 사무직, 전문직 및 기타)
  • Relationship: 0~5(각각 남편, 가족 관계 아님, 자녀, 독신, 부인 및 기타 친척)
  • Race: 0~4(각각 백인, 흑인, 아시아 및 태평양 섬 주민, 미국계 인디언 또는 에스키모, 기타 인종)
  • Capital gain: 0~80,000(USD)
  • Capital loss: 0~80,000(USD)
  • Hours worked per week: 0~100(시간)
  • Male?: 0~1(아니요 또는 예)
  • Female?: 0~1(아니요 또는 예)

참고: 다음 값 집합 중 하나를 이 순서대로 사용하여 웹 애플리케이션 양식을 채울 수 있습니다.

    • 27, 0, 4, 3, 1, 1, 2, 0, 0, 0, 40, 1, 0

    • 47, 0, 2, 2, 0, 0, 0, 0, 7688, 0, 44, 1, 0
