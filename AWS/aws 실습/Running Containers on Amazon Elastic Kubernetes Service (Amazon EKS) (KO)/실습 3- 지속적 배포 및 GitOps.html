실습 개요

지속적 전달(CD)은 개발자가 빠른 주기로 소프트웨어 업데이트를 자주 생성하고 수동으로 배포하도록 돕는 소프트웨어 개발 방식입니다. 이를 통해 자동화된 테스트 및 배포를 사용하고, 필요한 경우 실패한 변경 사항을 롤백하여 변경 사항을 안전하게 릴리스할 수 있습니다. 전달은 일반적으로 배포 파이프라인을 통해 이루어집니다. 파이프라인은 해당 코드를 컴파일, 테스트, 배포하기 위한 워크플로 및 도구에 대한 가시성을 제공합니다. 이 실습에서는 GitOps 선언적 파이프라인을 정의하고, 컨테이너 소스 파일 및 Kubernetes 매니페스트를 소스 제어 리포지토리에 저장하고, GitOps가 인프라 상태를 코드에 선언된 상태와 일치하도록 수렴하여 구성 드리프트를 방지하는 방법을 정의하는 방법을 보여줍니다.

AnyCompany 서비스 팀의 일원인 여러분은 Amazon Elastic Kubernetes Service(Amazon EKS) 및 Kubernetes 인프라를 위한 지속적 통합(CI) 및 지속적 전달(CD) 파이프라인을 배포하고 유지 관리하는 업무를 담당합니다. 여러분은 두 개의 파이프라인을 생성합니다. 첫 번째 파이프라인은 AWS CodeCommit, Jenkins 및 Amazon Elastic Container Registry(Amazon ECR)를 사용하여 컨테이너 이미지를 빌드하고 저장합니다. 두 번째 파이프라인은 AWS CodeCommit, Argo CD 및 Amazon EKS를 사용하여 컨테이너 이미지를 Amazon EKS 클러스터에 지속적으로 배포합니다.

목표

이 실습을 마치면 다음을 수행할 수 있습니다.

  • AWS CodeCommit 리포지토리를 생성 및 사용하여 GitOps 모범 사례에 따라 애플리케이션 코드와 Kubernetes 매니페스트 파일을 저장합니다.
  • Jenkins를 사용하여 지속적 통합 파이프라인을 구성합니다.
  • Kubernetes용 Argo CD 지속적 전달 도구를 배포합니다.
  • Argo CD를 사용하여 지속적 배포 파이프라인을 구성합니다.

과제 2: 실습용 배스천 호스트에 연결하고 AWS CodeCommit 및 Amazon ECR 연결을 위한 SSH 키 검토

이 과제에서는 실습 전반에서 사용하는 Amazon Elastic Compute Cloud(Amazon EC2) 인스턴스에 연결합니다. 그런 다음 자동으로 프로비저닝된 SSH 키를 검토하고 이를 gitUser라는 AWS Identity and Access Management(IAM) 사용자에 연결합니다. 또한 ecrUser라는 두 번째 IAM 사용자에 대해 액세스 키가 생성되었는지 확인합니다.

  1. AWS Management Console 상단의 검색 창에서

    Copied
    를 검색하여 선택하고 새 브라우저 탭에서 엽니다.

  2. EC2 Dashboard 의 Resources 섹션에서 Instances (running) 를 선택합니다.

  3. Bastion Host 인스턴스를 선택한 다음 Connect를 선택합니다.

  4. Connect to instance 페이지에서 Session Manager 탭을 선택합니다.

자세히 알아보기: AWS Systems Manager 기능인 Session Manager를 사용하면 방화벽 또는 Amazon Virtual Private Cloud(Amazon VPC) 보안 그룹에서 특정 포트를 열 필요 없이 배스천 호스트 인스턴스에 연결할 수 있습니다. 자세한 내용은 추가 자료 섹션의 AWS Systems Manager Session Manager를 참조하십시오.

  1. Connect를 선택합니다.

예상 출력: 배스천 호스트 인스턴스에 대한 연결이 완료된 새 브라우저 탭 또는 창이 열립니다. 일련의 시작 스크립트가 백그라운드에서 실행되어 후속 과제에서 사용되는 환경 변수를 설정합니다.

예상 출력:

Copied 
************************
**** EXAMPLE OUTPUT ****
************************

/* Jenkinsfile */
def imageTag
node {

    def app

    stage('Clone repository') {
        /* Let's make sure we have the repository cloned to our workspace */

        checkout scm
    }

    stage('Build image') {
        /* This builds the actual image; synonymous to
         * docker build on the command line */

        app = docker.build("eks-gitops-demo")
    }

    stage('Test image') {
        /* Run a test framework against our image. */

        app.inside {
            sh 'echo "Tests passed"'
        }
    }

    stage('Push image') {
        /* Finally, we'll push the image with two tags:
         * First, the incremental build number from Jenkins
         * Second, the 'latest' tag.
         * Pushing multiple tags is cheap, as all the layers are reused. */
        imageTag = sh(script: "head -n 1 Dockerfile | sed 's/#//'", returnStdout: true).trim()
        docker.withRegistry('https://ACCOUNTID.dkr.ecr.REGION.amazonaws.com', 'ecr:REGION:ecrUser') {
        /*    app.push("${env.BUILD_NUMBER}") */
            app.push("${imageTag}")
            app.push("latest")
        }
        echo "Pushed image with tag: ${imageTag}"
    }
}

참고: 출력에서 볼 수 있듯이 파이프라인은 4개 단계를 거칩니다.

  • Clone repository 단계에서는 checkout scm 명령을 사용하여 AWS CodeCommit에 인증하고 애플리케이션 코드를 Jenkins 작업 공간의 임시 디렉터리에 복제합니다.
  • Build image 단계에서는 Docker를 사용하여 새 eks-gitops-demo 컨테이너 이미지를 빌드합니다.
  • 모범 사례로 이미지를 리포지토리에 푸시하기 전에 자동화된 단위 테스트를 수행하는 것이 가장 좋습니다. 단위 테스트가 실패할 경우 파이프라인이 정상적으로 실패해야 하고 이미지가 리포지토리에 추가되면 안 됩니다. 그러나 이 실습에서는 콘솔에 Tests passed를 출력하는 간단한 스크립트만 실행합니다.
  • 마지막으로, Push image 단계에서는 Jenkins용 Amazon ECR 플러그 인을 사용하여 테스트된 이미지를 ECR에 푸시합니다. 이미지에는 Git SHA 및 latest 태그가 지정됩니다. 계층이 재사용되므로 여러 태그를 푸시하는 것이 저렴합니다.

이 워크플로는 다음을 보장합니다.

  • 이미지가 최신 코드에서 빌드됩니다.
  • 자동화된 테스트가 이미지의 유효성을 검사합니다.
  • 테스트가 통과한 경우에만 이미지가 ECR로 푸시됩니다.

주의: Push image 단계에서 Amazon ECR URL 및 ecrUser 자격 증명에 자리 표시자 텍스트가 포함되어 있습니다. 이들 값을 본인의 계정 ID 및 AWS 리전으로 바꿔야 합니다.

과제 6: Argo CD 설치 및 AWS CodeCommit 연결 설정

이 과제에서는 Argo CD를 Amazon EKS의 자체 네임스페이스에 설치하고, Argo CD UI에 액세스하고, AWS CodeCommit과의 연결을 구성합니다.

과제 6.1: Argo CD 설치

Argo CD는 Kubernetes를 위한 선언형 GitOps 지속적 전달 도구입니다. 이 도구는 원하는 애플리케이션 상태를 정의하기 위한 신뢰할 수 있는 정보 소스로 Git 리포지토리를 사용하는 GitOps 패턴을 따릅니다. Argo CD는 대상 환경에서 원하는 애플리케이션 상태의 배포를 자동화합니다. 애플리케이션 배포는 브랜치 및 태그에 대한 업데이트를 추적하거나 Git 커밋 시 특정 버전의 매니페스트에 고정될 수 있습니다.

Argo CD는 실행 중인 애플리케이션을 지속적으로 모니터링하고 현재 실시간 상태를 (Git 리포지토리에 지정된) 원하는 상태와 비교하는 Kubernetes 컨트롤러로 구현됩니다. 실시간 상태가 목표 상태에서 벗어나는 배포된 애플리케이션은 OutOfSync로 간주됩니다. Argo CD는 차이점을 보고 및 시각화하는 동시에 실시간 상태를 원하는 목표 상태로 자동 또는 수동으로 다시 동기화하는 기능을 제공합니다. Git 리포지토리에서 수정된 원하는 목표 상태는 지정된 대상 환경에 자동으로 적용되고 반영될 수 있습니다.

과제 8.5: Argo CD에서 업데이트된 애플리케이션 배포 확인

이 과제에서는 Argo CD가 Kubernetes 배포를 자동으로 업데이트했고 Pod가 새 애플리케이션 버전을 실행하고 있는지 확인합니다.

  1. Argo CD Applications 페이지가 열려 있는 브라우저 탭으로 돌아갑니다.

  2. Applications > eks-sample-linux-app 페이지의 REFRESH 드롭다운 목록에서 HARD REFRESH를 선택합니다.

현재 새 버전의 애플리케이션이 실행 중이고 Argo CD가 자동으로 업데이트되었는지 확인합니다.

참고: Argo CD의 SYNC 작업은 애플리케이션의 실시간 상태를 Git에 정의된 원하는 목표 상태로 동기화하는 데 사용됩니다. 이 옵션은 프로세스 속도를 높이려는 경우에만 사용하십시오.

SYNC 작업을 실행하면 다음을 수행합니다.

  • Git 리포지토리에 연결하고 최신 커밋을 가져옵니다.
  • 지정된 대상 매니페스트를 클러스터에 포함된 리소스의 실시간 상태와 비교합니다.
  • Git에 지정된 원하는 상태와 실시간 상태 간의 차이점, 즉 드리프트를 감지합니다.
  • Git의 대상 매니페스트와 일치하도록 클러스터의 리소스를 수정합니다.

이를 통해 클러스터의 애플리케이션 구성이 Git 리포지토리에 정의된 구성과 일치하게 됩니다.
