실습 개요

Kubernetes는 컨테이너식 애플리케이션의 배포, 확장 및 관리를 자동화하기 위한 오픈 소스 시스템입니다. Amazon Elastic Kubernetes Service(Amazon EKS)는 자체 Kubernetes 제어 영역을 유지 관리할 필요 없이 AWS에서 Kubernetes를 실행하는 데 사용할 수 있는 관리형 서비스입니다.

여러분은 Example Corp의 컨테이너 관리자로서 컨테이너 애플리케이션 및 인프라의 배포 및 유지 관리를 담당합니다. Example Corp은 Amazon EKS에서 Kubernetes 애플리케이션을 구현하기로 결정했습니다. 여러분은 kubectl 유틸리티를 사용하여 Kubernetes 리소스를 구현하고 액세스할 책임이 있습니다.

이 실습에서는 kubectl을 사용하여 Amazon EKS 클러스터에 액세스합니다. Kubectl은 Kubernetes API 서버와 통신하기 위한 명령줄 인터페이스(CLI) 유틸리티입니다. 이 유틸리티를 사용하여 Kubernetes 애플리케이션 리소스를 구현하고, 리소스 세부 정보를 가져오고, Kubernetes Pod에서 직접 실행하고, Kubernetes 객체를 삭제할 수 있습니다. 그런 다음 클러스터에 Product Catalog 애플리케이션을 배포하고 활성 및 준비 상태 프로브를 구성합니다.

목표

이 실습을 마치면 다음을 수행할 수 있습니다.

  • Kubernetes 애플리케이션을 생성하고 배포합니다.
  • 배포, 서비스 및 네임스페이스 리소스를 빌드합니다.
  • 네임스페이스에서 리소스를 봅니다.
  • 서비스 및 배포의 세부 정보를 분석합니다.
  • Pod의 세부 정보를 이해합니다.
  • Pod에서 명령을 실행합니다.
  • 활성 및 준비 상태 프로브를 구현합니다.
  • 애플리케이션을 삭제합니다.

참고: 보다시피 클러스터에는 5개의 네임스페이스가 채워져 있습니다.

  • default: 네임스페이스를 지정하지 않으면 새로 생성된 Kubernetes 객체가 이 네임스페이스에 배포됩니다.

  • kube-system: 이 네임스페이스에는 Kubernetes에서 생성한 객체가 포함되어 있습니다.
  • workshop: 이 네임스페이스는 AWS CloudFormation에서 생성한 것이며 애플리케이션을 보관하는 데 사용됩니다.

마이크로서비스 기술 기능
프런트엔드 Nodejs 및 EJS 템플릿 작성 언어 프런트엔드 서비스는 애플리케이션 UI를 표시합니다.
Product Catalog Python Flask Restplus 및 Swagger UI 카탈로그에 제품을 추가하고 제품 세부 정보를 검색합니다.
Catalog Detail NodeJS 공급업체 이름 및 버전 번호를 반환합니다.

프런트엔드 마이크로서비스와 백엔드 마이크로서비스 중 하나는 이미 클러스터에 배포되어 있습니다. 다음 단계에서는 Catalog Detail 마이크로서비스를 배포하고 노출합니다.

  1. 명령: Catalog Detail 마이크로서비스를 배포하려면 먼저 프런트엔드 배포를 설명하는 매니페스트부터 생성합니다. 매니페스트를 생성하려면 다음 명령을 입력합니다.
Copied 
cat << EOF > ~/proddetail-deployment.yaml
apiVersion: apps/v1
kind: Deployment
metadata:
  name:  proddetail
  namespace: workshop
spec:
  replicas: 1
  selector:
    matchLabels:
      app: proddetail
  template:
    metadata:
      labels:
        app: proddetail
    spec:
      containers:
        - name: proddetail
          image: "public.ecr.aws/u2g6w7p2/eks-workshop-demo/catalog_detail:1.0"
          imagePullPolicy: Always
          ports:
            - name: http
              containerPort: 3000
              protocol: TCP
          resources: {}
EOF

참고: Kubernetes 매니페스트는 YAML 또는 JSON으로 작성될 수 있으며 선언형 프로그래밍의 예입니다. 각 명령을 실행하는 방법을 명시하는 명령형 프로그래밍과 달리 선언형 프로그래밍은 원하는 애플리케이션 상태를 설명합니다. 그런 다음 선언형 프로그래밍 언어는 원하는 상태에 도달하는 데 필요한 논리를 배포합니다.

방금 생성한 배포 매니페스트는 proddetail 배포의 원하는 상태를 설명합니다. 이 경우에는 매니페스트가 퍼블릭 Amazon ECR 리포지토리의 컨테이너 이미지를 사용하여 하나의 Pod가 포함된 ReplicaSet를 생성합니다. 매니페스트 구성 요소를 자세히 살펴보겠습니다.

  • 이 매니 페스트는 v1 APIVersion을 참조하여 Kubernetes 배포 객체를 생성합니다. 배포는 매니페스트에 정의된 애플리케이션 상태를 유지하려고 시도할 때 Pod를 동적으로 생성 및 삭제하는 영구 객체입니다.
  • metadata 레이블은 배포에 이름을 할당하고 배포가 배포될 네임스페이스를 선택하는 데 사용됩니다. 네임스페이스를 선택하지 않으면 배포가 default 네임스페이스에 배포됩니다.
  • kind 레이블은 생성하려는 Kubernetes 객체의 유형을 선택하고 spec 레이블은 해당 상태를 정의합니다.
  • 이 매니페스트는 ReplicaSet를 생성합니다. ReplicaSet는 특정 수의 Pod 인스턴스가 항상 실행되고 있는지 확인하는 데 사용됩니다. 이 경우에는 매니페스트가 복제본 1개를 정의합니다. 즉, API는 하나의 Pod 인스턴스가 항상 실행되도록 시도합니다. replicas 필드를 3으로 변경하면 두 개의 추가 Pod가 생성되고 유지 관리됩니다.
  • selector 필드는 Kubernetes에게 ReplicaSet가 획득해야 할 Pod를 알려줍니다. 이 ReplicaSet는 매니페스트에 app: proddetail 레이블이 포함된 모든 Pod에 적용됩니다.
  • template 필드는 새 Pod를 생성하는 데 사용해야 하는 데이터를 식별합니다. 이 템플릿은 퍼블릭 Amazon ECR 이미지를 사용하여 컨테이너를 생성하고, 이름을 proddetail로 지정하고, HTTP 트래픽에 대해 포트 3000을 개방합니다.
  1. 명령: Product Detail 서비스의 상태를 선언하는 두 번째 매니페스트를 생성합니다. 매니페스트를 생성하려면 다음 명령을 입력합니다.

과제 3: 활성 및 준비 상태 프로브 구현

이 과제에서는 활성 및 준비 상태 프로브를 포함하도록 Product Detail 배포를 업데이트한 다음, 의도적으로 오류를 주입하여 테스트합니다.

참고: 장기간 실행되는 애플리케이션은 대부분 중단 상태로 전환되며 다시 시작하지 않으면 복구할 수 없습니다. Kubernetes는 상태 확인을 사용하여 이러한 상황을 감지하고 수정합니다. 예를 들어, 활성 프로브를 사용하여 애플리케이션이 실행 중이지만 진행할 수 없는 교착 또는 경합 상태를 포착할 수 있습니다. 이러한 상태에서 컨테이너를 다시 시작하면 버그에도 불구하고 애플리케이션의 가용성을 높이는 데 도움이 될 수 있습니다. 마찬가지로, 준비 상태 프로브는 서비스가 트래픽 수신을 시작할 준비가 되지 않은 Pod에 요청을 보내지 않도록 하는 데 도움이 됩니다.

그러나 활성 프로브와 준비 상태 프로브를 모두 사용하여 애플리케이션을 구성하는 경우 활성 프로브는 준비 상태 프로브가 성공할 때까지 기다리지 않습니다. 따라서 관리자는 활성 프로브에서 initialDelaySeconds 파라미터를 활성화하거나 시작 프로브를 구성하는 것이 좋습니다.

  1. 명령: 활성 및 준비 상태 프로브가 포함된 배포 매니페스트는 BastionHost에 저장되어 있습니다. 이 매니페스트를 살펴보기 위해 다음 명령을 입력합니다.
Copied 
cat -n ~/detail_deployment_with_probes.yaml && echo

예상 출력:

Copied 
************************
**** EXAMPLE OUTPUT ****
************************

     1  apiVersion: apps/v1
     2  kind: Deployment

periodSeconds 필드는 kubelet이 배포를 프로브해야 하는 주기를 설정합니다. 이 예에서는 kubelet이 5초마다 활성 프로브를 확인하고 3초마다 준비 상태 프로브를 확인합니다. 또한 두 프로브 모두에 새로운 Pod가 생성된 후 kubelet이 첫 번째 프로브를 수행하기 전에 대기해야 하는 시간을 나타내는 initialDelaySeconds 필드가 포함되어 있습니다.

Catalog Detail 마이크로서비스는 명령 프로브를 사용하도록 구성되었습니다. 이 프로브는 readiness.txt라는 텍스트 파일에서 ready와 일치하는 문자열을 검색하는 간단한 Linux 명령을 수행합니다. 명령이 성공적으로 실행되고 종료 코드 0을 반환하면 Pod가 정상 상태로 간주됩니다. 반면, 프로브가 연속 3번 실패하면 프로브가 실패로 간주되어 Pod가 종료됩니다.
