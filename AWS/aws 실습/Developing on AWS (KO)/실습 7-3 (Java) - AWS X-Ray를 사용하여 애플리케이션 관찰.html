개요

이전 실습에서 구축한 애플리케이션이 AWS SAM으로 다시 배포되었습니다. AWS SAM은 서버리스 애플리케이션을 구축하기 위한 오픈 소스 프레임워크입니다. 함수, API, 데이터베이스 및 이벤트 소스 매핑을 표현하는 약식 구문을 제공합니다. AWS X-Ray로 애플리케이션 코드를 계측하여 애플리케이션의 작동 상태를 관찰합니다.

전체 애플리케이션은 아래 다이어그램에 표시됩니다.

  • AWS X-Ray 기능을 사용하도록 애플리케이션 코드를 계측합니다.
  • 애플리케이션 배포 패키지를 활성화하여 로그를 생성합니다.
  • AWS SAM 템플릿의 주요 구성 요소를 이해하고 애플리케이션을 배포합니다.
  • X-Ray Trace map을 생성하여 애플리케이션의 엔드 투 엔드 처리 동작을 관찰합니다.
  • X-Ray 트레이스 및 주석을 사용하여 애플리케이션 문제를 분석하고 디버깅합니다.

과제 1: 코드에서 AWS X-Ray 기능 구성

Java용 X-Ray SDK는 X-Ray 데몬(daemon)에 트레이스 데이터를 생성하고 전송하기 위한 클래스와 메서드를 제공하는 Java 웹 애플리케이션용 라이브러리 세트입니다. 트레이스 데이터에는 애플리케이션에서 제공하는 수신 HTTP 요청에 관한 정보가 포함되며, 애플리케이션이 AWS SDK, HTTP 클라이언트 또는 SQL 데이터베이스 커넥터를 사용하여 다운스트림 서비스에 대해 수행하는 호출도 포함됩니다. 세그먼트를 수동으로 생성하고 주석 및 메타데이터에 디버그 정보를 추가할 수도 있습니다.

X-Ray는 서비스 통합을 사용하여 Lambda, DynamoDB, API Gateway 등 지원되는 서비스에서 트레이스 데이터를 수신합니다. Java용 AWS X-Ray SDK는 코드에서 트레이스 데이터를 전송하는 데 사용됩니다.

추가 정보: 모든 코드는 AWS SAM으로 관리 및 배포됩니다. 이제 AWS Cloud9 환경에서 이 코드에 액세스할 수 있습니다.

고려 사항: 이 실습은 숙련된 개발자와 신입 개발자 모두를 위해 고안되었습니다.

  • 도전을 즐기는 숙련된 개발자를 위해 각 과제에 앞서 과제 완료에 도움이 되는 충분한 정보가 담긴 개략적 지침이 제공됩니다.

  • 업데이트를 완료하면 코드를 테스트하여 작동하는지 확인하고, 필요시 문제를 해결한 후 다음 과제를 진행하십시오.

  • 신입 개발자는 실습의 각 단계를 자세히 설명하는 상세 지침을 참고할 수 있습니다.

개략적 지침

  • 버전 2.9.0의 aws-xray-recorder-sdk-bom 종속성을 Maven 빌드 구성에 추가하여 확인합니다.
  • AWS X-Ray에 필요한 라이브러리를 delete-note 함수 코드로 가져옵니다.
  • delete-function 코드에서 userId 및 noteId를 사용해 AWS X-Ray 주석을 삽입합니다.
  • Java용 AWS SDK 클라이언트를 통해 이루어진 모든 서비스 호출 요청에 대해 AWS X-Ray 로깅을 활성화합니다.

과제 2.1: AWS SAM 템플릿 사용자 지정 및 구성 요소 식별

AWS SAM 템플릿은 AWS CloudFormation 템플릿의 확장에 속하며, 작업을 더 쉽게 처리하는 몇 가지 추가 구성 요소가 있습니다.

  1. api/template.yml을 열어 AWS SAM 템플릿을 봅니다.

SAM 템플릿은 Transform 선언인 Transform: AWS::Serverless-2016-10-31로 시작합니다. 이 선언은 AWS CloudFormation 템플릿 파일을 AWS SAM 템플릿으로 식별합니다.

Globals 섹션은 AWS SAM에서만 볼 수 있습니다. 이 섹션은 모든 서버리스 함수와 API에 공통적인 속성을 정의합니다. AWS::Serverless 유형의 모든 CloudFormation 리소스는 Globals 섹션에 정의된 속성을 상속합니다. 이 템플릿에서는 API Gateway 리소스에 대한 로깅이 활성화되었습니다.

AWS SAM 템플릿에서 Resources 섹션은 AWS CloudFormation 리소스와 AWS SAM 리소스의 조합을 포함할 수 있습니다. 이 템플릿은 서버리스 리소스를 먼저 정의합니다.

pollyNotesAPI 리소스는 애플리케이션에서 사용할 API Gateway 리소스를 정의합니다. 여기에서 Prod 단계가 설정되고 CORS가 활성화되며 리전 배포 유형이 지정되고 사용할 권한 부여자가 지정됩니다. CORS는 Lambda 프록시 통합을 사용하지 않고 생성할 수 있는 OPTIONS 메서드 또는 그 밖의 메서드에 대해 활성화됩니다.

5개의 AWS Lambda 함수는 AWS::Serverless::Function 리소스를 사용하여 정의됩니다. 각 리소스는 템플릿을 기준으로 코드의 위치를 지정합니다. 그런 다음, 리소스는 핸들러 함수, 런타임, AWS Identity and Access Management(IAM) 역할 및 이벤트 트리거를 정의합니다. 이러한 함수는 하나의 계층으로 패키징된 공통 라이브러리를 참조합니다. 해당 계층은 AWS::Serverless::LayerVersion을 사용하여 정의되며 공통 종속성은 makefile을 사용하여 빌드됩니다.

리소스의 Events 섹션은 AWS::Serverless::Function을 트리거하는 이벤트를 지정합니다. 이벤트는 하나의 유형과 이 유형에 따라 달라지는 속성 집합으로 구성됩니다. 이 템플릿에서는 요청을 함수로 라우팅할 API Gateway 리소스에 대한 메서드를 생성합니다. 권한 부여자를 지정하면 인증된 요청만 이 메서드에 액세스할 수 있습니다. Lambda 프록시 통합으로 인해 API 구성을 위한 함수 코드가 이전 실습보다 훨씬 간단합니다.

함수 뒤에 있는 pollyNotesTable은 애플리케이션에 대한 데이터를 보관하는 테이블입니다. 이 템플릿은 정렬 키를 필요로 하는 일반적인 AWS::DynamoDB::Table 리소스를 사용합니다. CloudFormation 템플릿에서 파티션 키인 UserId는 HASH로, 정렬 키인 NoteId는 RANGE로 정의됩니다. 요금제는 온디맨드 방식인 요청 별 비용청구 방식으로 설정됩니다. 애플리케이션이 사용할 트래픽의 양을 더 잘 파악한 후 타당하다면 이를 프로비저닝 모드로 변경할 수 있습니다.

다음 4개 리소스는 애플리케이션이 인증을 처리하도록 Amazon Cognito를 구성합니다.

마지막으로 Outputs 섹션에서는 이 API를 사용하도록 웹 애플리케이션을 구성하는 데 필요한 값을 사용할 수 있습니다.

과제 5.1: 올바른 IAM 역할을 delete 함수와 연결

  1. ‘Cloud9-…’ 웹 브라우저 탭의 api 폴더에서 template.yml을 엽니다.

Lambda 함수를 생성할 때 AWS 리소스에 액세스하기 위한 IAM 역할을 지정해야 합니다. 이 실습에서 사용할 세 가지 IAM 역할이 생성되어 있습니다.

  • DynamoDBReadRole
  • DynamoDBWriteRole
  • DictateRole

모든 역할은 함수가 Amazon CloudWatch 및 AWS X-Ray에 쓰기를 할 수 있도록 허용합니다. 첫 두 가지 역할은 이 애플리케이션의 DynamoDB 테이블에 대한 읽기 또는 쓰기 액세스를 허용합니다. 마지막 역할은 dictate 함수에만 사용되며 DynamoDB, S3 및 Polly에 액세스하는 데 필요한 권한을 제공합니다. deleteFunction 리소스를 보면 잘못된 역할이 구성되었습니다. 잘못된 역할 ARN을 올바른 ARN으로 바꿉니다.

  1. deleteFunction 내에 Lambda 역할 속성을 찾아서 다음과 같이 변경 합니다.- !Sub arn:aws:iam::${AWS::AccountId}:role/DynamoDBReadRole 에서

    Copied
    으로 변경합니다.

  2. template.yml을 저장하고 닫습니다.

선택 과제 5.4: AWS X-Ray에서 애플리케이션 트래픽 검토

AWS X-Ray는 활동이 있을 때마다 애플리케이션에서 새로운 트레이스를 계속 수신합니다. X-Ray 콘솔로 돌아가서 이제 모든 것이 정상인지 확인합니다.

  1. 로그아웃했다가 다시 웹 애플리케이션에 로그인하고 과제 3.2에서 수행한 것과 동일한 테스트 작업을 반복합니다.

  2. ‘AWS X-Ray’ 웹 브라우저 탭의 왼쪽 탐색 패널에서 Trace Map을 선택하고 애플리케이션의 상태를 확인합니다.

요약

축하합니다! Trace map에는 최근 테스트에 대한 성공적인 트레이스만 표시됩니다. X-Ray로 전송된 트레이스 데이터는 일반적으로 서비스에서 이를 수신한 후 30초 이내에 검색 및 필터링할 수 있습니다. X-Ray는 지난 30일간의 트레이스 데이터를 저장합니다. 따라서 30일 전의 트레이스 데이터를 쿼리할 수 있습니다.

이제 다음을 수행할 수 있습니다.

  • AWS X-Ray 기능을 사용하도록 애플리케이션 코드를 계측합니다.
  • 애플리케이션 배포 패키지를 활성화하여 로그를 생성합니다.
  • AWS SAM 템플릿의 주요 구성 요소를 이해하고 애플리케이션을 배포합니다.
  • X-Ray Trace map을 생성하여 애플리케이션의 엔드 투 엔드 처리 동작을 관찰합니다.
  • X-Ray 트레이스 및 주석을 사용하여 애플리케이션 문제를 분석하고 디버깅합니다.
