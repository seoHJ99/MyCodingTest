개요

이제 스토리지, 데이터베이스 및 컴퓨팅 처리를 위한 코어 백엔드가 설정되었으므로 사용자가 인터넷을 통해 AWS Lambda 함수에 액세스할 수 있는 방법이 필요합니다.

이 실습에서는 애플리케이션에 대한 Amazon API Gateway 리소스를 생성합니다. 그런 다음, 해당 리소스에서 두 가지 AWS Lambda 함수에 대한 액세스를 허용하는 두 가지 방법을 구성합니다. 이러한 방법을 사용하여 요청을 검증하고 Lambda 함수에서 반환된 응답을 변환합니다. 마지막으로 다음 실습에서 추가될 웹 애플리케이션에 대한 액세스를 허용하도록 CORS(Cross-Origin Request Sharing) 헤더를 구성합니다.

이 실습을 완료하면 배포는 다음 다이어그램과 같이 됩니다.

  • API를 Stage(단계)에 배포하고 API 엔드포인트를 사용하여 결과를 검증합니다.

과제 2: GET 메서드를 구성하고 응답 변환을 위한 매핑 템플릿을 추가

액세스를 허용해야 하는 첫 번째 Lambda 함수는 list-function입니다. GET 메서드를 사용하여 API Gateway 리소스에 요청을 보낼 때마다 이 함수가 호출되고 응답이 요청자에게 반환되어야 합니다. 이벤트에 UserId가 지정된 경우 Lambda 함수는 해당 UserId와 일치하는 항목만 반환합니다. 통합 요청을 사용하여 지금은 해당 파라미터를 하드 코딩하는 매핑 템플릿을 생성합니다.

단일 사용자에 대한 항목을 반환하도록 요청을 구성한 후에는 반환되는 모든 항목에 UserId를 포함시킬 필요가 없습니다. 애플리케이션이 확장됨에 따라 성능을 높이고 데이터 전송 비용을 줄이려면 필요한 정보가 포함된 응답만 반환하는 것이 중요합니다.

개략적 지침

  • 노트 리소스에 대한 GET 메서드를 구성하고 이를 list-function에 매핑한 후 출력을 테스트합니다.
  • 매핑 템플릿을 추가하여 수동으로
    Copied
    를 지정하고 출력을 확인합니다.
  • 노트 ID와 노트 텍스트만 반환하도록 응답을 변환하는 응답 매핑 템플릿을 생성합니다.

과제 2.2: UserId 변수를 전달하도록 GET 메서드 통합 요청 업데이트

통합 요청은 API Gateway에서 REST API 메서드의 내부 인터페이스입니다. 통합 요청에서 경로 요청의 본문 또는 메서드 요청의 파라미터 및 본문을 백엔드에 필요한 형식으로 매핑합니다.

추가 정보: 매핑 템플릿은 요청 또는 응답 본문을 필요한 데이터 형식으로 변환하는 VTL(Velocity Template Language)의 스크립트입니다. 통합 요청 또는 통합 응답에 매핑 템플릿이 지정될 수 있습니다. 이 템플릿은 런타임에서 컨텍스트 및 Stage(단계) 변수로 사용할 수 있는 데이터를 참조할 수 있습니다.

매핑은 클라이언트에서 백엔드로 요청을 하던 것처럼 Integration을 통해 헤더나 본문을 전달하는 것이 자격증명 변환만큼 간단할 수 있습니다. 백엔드에서 클라이언트로 페이로드가 전달되는 응답의 경우도 마찬가지입니다.

    을 입력합니다.

  1. 페이지 상단의 /notes - GET - Method Execution 패널에서 Test 탭을 선택합니다.

이제 응답은 각 객체에 NoteId 및 Note 속성만 있는 JSON 어레이입니다. 이 응답에는 UserId가 필요하지 않으므로 반환되는 데이터의 양을 줄일 수 있습니다. 더 많은 노트가 시스템에 입력됨에 따라 응답 크기가 기하급수적으로 줄어듭니다. 응답이 적을수록 애플리케이션 성능이 향상되고 데이터 전송 비용이 절감됩니다.

과제 3: POST 메서드 구성 및 요청 검증 모델 추가

이제 데이터베이스에서 데이터를 검색할 수 있습니다. 데이터베이스에서 데이터를 추가하거나 업데이트하는 방법도 필요합니다. 액세스를 제공할 다음 Lambda 함수는 createUpdate-function입니다. 이 기능에 액세스하려면 POST 메서드를 사용합니다. 노트 리소스에 새 노트를 게시할 때에는 추가(또는 업데이트)할 데이터를 요청 본문에 포함합니다.

기본값 설정에 따라 전체 본문은 그대로 백엔드에 전달됩니다. API Gateway의 특징 중 하나는 모델을 사용하여 데이터 구조를 확인할 수 있다는 것입니다.

모델은 요청 또는 응답 페이로드의 데이터 구조를 지정하는 데이터 스키마입니다. 모델은 API에서 강력한 형식의 SDK를 생성하는 데 필요합니다. 또한 페이로드를 확인하는 데에도 사용됩니다.

백엔드로 전달되기 전에 데이터 구조를 확인하여 Lambda 함수를 호출하기 전에 입력 형식이 올바른지 확인할 수 있습니다. 입력이 모델 스키마와 일치하지 않으면 API Gateway는 즉시 오류 메시지로 응답하고 Lambda 함수를 호출하지 않습니다.

개략적 지침

  • POST 메서드를 구성하여 createUpdate-function에 매핑하고 새 노트 객체로 기본 동작을 테스트합니다.
  • 위의 노트 스키마를 지정하는 모델을 생성합니다. (힌트가 필요한 경우 과제 3.2에서 스키마를 찾을 수 있습니다.)
  • 요청 본문에 관한 검증을 구성하고 스키마를 적용하며 유효하거나 잘못된 제출을 테스트합니다.

과제 4: CORS 구성으로 API 배포

CORS(Cross-Origin Resource Sharing)는 브라우저에서 리소스 로드를 허용해야 할 자체 오리진이 아닌 다른 오리진(domain, scheme 또는 port)을 서버가 나타낼 수 있도록 하는 HTTP 헤더 기반 메커니즘입니다. 또한 CORS는 서버가 실제 요청을 허용하는지 확인하기 위해 브라우저가 교차 오리진 리소스를 호스트하는 서버에 ‘프리플라이트’ 요청을 하는 메커니즘에 의존합니다. 이 프리플라이트에서 브라우저는 HTTP 메서드를 나타내는 헤더와 실제 요청에 사용될 헤더를 전송합니다.

교차 오리진 요청의 예: https://domain-a.com에서 제공되는 프런트엔드 JavaScript 코드는 XMLHttpRequest를 사용하여 https://domain-b.com/data.json을 요청합니다.

이후 실습에서는 Amazon S3 도메인 이름을 사용하여 웹 애플리케이션에 액세스하고 API에는 Amazon API Gateway 도메인 이름을 사용하므로 실습에서 이러한 구성이 필요합니다. 다른 도메인으로 백엔드 API를 호스트하는 데는 여러 가지 이유가 있으며 CORS 구성은 API 개발의 공통 요구 사항에 속합니다.

CORS를 구성한 후 새 Stage(단계)를 생성하고 API를 배포합니다. 이렇게 하면 인터넷에서 액세스할 수 있는 엔드포인트가 생성됩니다.

개략적 지침

  • 모든 오리진을 허용하려면 모든 메서드에서 CORS를 활성화합니다.
  • API를 Prod Stage(단계)에 배포합니다.
  • 웹 브라우저로 API를 테스트합니다.

상세 지침

과제 4.1: CORS(Cross-Origin Resource Sharing) 구성

API를 처음 생성할 때 CORS를 활성화하도록 서비스에 지시했습니다. 리소스에 대한 OPTIONS 메서드를 생성하고 구성했습니다. OPTIONS 메서드는 브라우저가 ‘프리플라이트’ 요청을 수행할 때 요청됩니다. 메서드를 생성했으므로 해당 메서드에 대해 CORS를 구성해야 합니다.

이 과제에서는 구성한 메서드에 Response Headers를 추가합니다. 이는 수동으로 구성할 수 있지만 아래 지침에서는 API Gateway 콘솔을 사용하여 구성을 자동화합니다.

  1. Resources 패널에서 /notes를 선택합니다.

  2. Choose the Enable CORS .

  • Gateway Responses에서 Default 4XX 및 Default 5XX의 확인란을 선택합니다. 이는 문제 해결이 필요한 경우 브라우저에서 4xx 및 5xx 오류를 보기 위한 것입니다.

  • Access-Control-Allow-Methods에서 GET, OPTIONS 및 POST 확인란을 선택합니다.

  1. Save를 선택합니다.

각 기존 메서드에 대해 이 단계에서는 메서드가 구성한 각 상태 코드에 대해 Header Mappings 및 Response Headers를 추가했습니다. 이제 웹 브라우저에서 이러한 API 메서드에 대해 CORS를 허용합니다. 웹 프런트엔드 및 API가 알려진 도메인 이름을 사용할 때 향후 보안을 강화하도록 이 구성을 수정할 수 있습니다.

과제 4.2: API 배포

API가 구성되었지만 API 엔드포인트로 이 구성에 액세스하려면 먼저 Stage(단계)를 생성하고 이 Stage(단계)에 구성을 배포해야 합니다.

Stage(단계)는 API의 수명 주기 상태에 대한 논리적 참조입니다. API Stage(단계)는 API ID와 Stage(단계) 이름으로 식별됩니다. ‘dev’ 및 ‘prod’ Stage(단계)를 생성할 수 있으며 모든 변경 사항을 dev에 배포한 후 이를 prod에 배포하기 전에 완전히 테스트할 수 있습니다.

또 다른 공통 전략은 API의 메이저 버전 또는 마이너 버전에 대해 새로운 Stage(단계)를 갖는 것입니다. API를 사용하는 프런트엔드 애플리케이션의 레거시 버전에 영향을 줄 수 있는 변경을 수행할 때마다 새 Stage(단계)(예: ‘v2’)를 생성할 수 있습니다. 사용자는 코드가 새 버전에서 작동하는지 확인한 후 새 기능을 활용하려면 API 엔드포인트 URL을 변경해야 합니다.

API 배포는 API Gateway API의 특정 시점 스냅샷입니다. 클라이언트가 사용할 수 있으려면 배포가 하나 이상의 API Stage(단계)와 연결되어야 합니다.

이 실습에서 사용하지 않는 AWS 서비스는 실습 환경에서 비활성화됩니다. 또한 이 실습에 사용되는 서비스의 기능은 실습에 필요한 작업으로 제한됩니다. 다른 서비스에 액세스하거나 실습 안내서에서 제공하는 것 외의 작업을 수행하는 경우 오류가 발생할 수 있습니다.

아이콘 설명

이 실습에서는 다양한 유형의 지침 및 참고 사항에 대한 주의를 환기하기 위해 다양한 아이콘이 사용됩니다. 모든 아이콘이 사용되지는 않지만 다음 목록에서는 각 아이콘의 용도를 설명합니다.

  • 명령: 실행해야 하는 명령입니다.
  • 예상 출력: 명령 또는 편집된 파일의 출력을 확인하는 데 사용할 수 있는 샘플 출력입니다.
  • 참고: 참고 사항, 팁 또는 중요한 가이드입니다.
  • 추가 정보: 자세한 정보를 찾을 수 있는 위치입니다.
  • 주의: 특별한 관심이 필요한 중요한 정보입니다(놓쳤다고 해도 장치 또는 데이터에 문제가 발생할 정도로 중요하지는 않지만 특정 Stage(단계)를 반복해야 할 수 있음).
  • 경고: 되돌릴 수 없고, 명령 또는 프로세스의 실패에 영향을 줄 수 있는 작업에 대해 주의하도록 합니다(구성을 설정한 후 변경할 수 없는 구성에 대한 경고가 포함됨).
  • 고려 사항: 자신의 환경에 개념을 적용하는 방법을 고려하거나 당면한 주제에 대한 대화를 시작하기 위해 일시 중지할 시점을 제안합니다.
  • 복사/붙여넣기: 복사 및 붙여넣기가 필요한 미리 생성된 스크립트 또는 파일의 내용을 표시하는 코드 블록입니다. 코드 블록의 특정 부분만 복사해야 하는 경우 코드에 번호가 매겨진 TODO 주석이 있습니다.
  • 이해도 점검: 지식을 확인하고 학습한 내용을 테스트하는 기회입니다.
  • 보안: 보안 모범 사례를 적용할 기회입니다.
  • 새로 고침: 웹 브라우저 페이지 또는 목록을 새로 고쳐 새 정보를 표시해야 하는 경우입니다.
  • 복사 명령: (특정 변수를 편집하기 위해) 명령, 스크립트 또는 기타 텍스트를 텍스트 편집기로 복사하는 것이 명령줄 또는 터미널에서 직접 편집하는 것보다 쉬울 수 있는 경우입니다.
  • 힌트: 질문 또는 도전 과제에 대한 힌트입니다.
  • 해답: 질문 또는 도전 과제의 해답입니다.
  • 그룹 활동: 과제를 완료하기 위해 다른 수강생과 함께 작업해야 하는 경우입니다.
