실습 개요

이 실습에서는 Identity and Access Management(IAM)가 자격 증명 기반 정책과 리소스 기반 정책 및 권한 경계를 사용하여 권한을 평가하는 방법을 보여 줍니다. IAM 역할 또는 AWS 리소스에 정책을 연결하는 방법과 이러한 정책 평가에 따라 효과적인 권한이 생성되는 방법을 알아봅니다. 그런 후 IAM 권한 경계를 사용하여 IAM 역할에 할당할 수 있는 권한에 대해 제한을 설정합니다. 이 실습에서 소개하는 개념은 IAM을 사용하여 단일 계정의 자격 증명 및 리소스를 관리하는 출발점 역할을 합니다.

목표

본 실습을 마치면 다음을 할 수 있습니다.

  • 정책을 통해 권한을 정의합니다.
  • 자격 증명 또는 리소스 기반 정책을 사용하여 권한을 관리합니다.
  • IAM 권한 경계를 사용하여 IAM 역할에 대한 최대 권한을 설정합니다.

필수 기술 지식

이 실습을 정상적으로 완료하려면 AWS Cloud Practitioner Essentials 교육 과정에서 정의한 AWS 관련 기본 지식을 갖춰야 합니다. AWS 관리 콘솔을 탐색하는 것과 기본 사용자 인증 개념에 익숙해야 합니다.

시나리오

여러분은 AnyCompany의 클라우드 보안 관리자로서 조직의 개발 환경 내 IAM 관리를 맡고 있습니다. AnyCompany의 개발자는 원격으로 작업하고 관리형 Amazon Elastic Compute Cloud(Amazon EC2) 인스턴스에 액세스하여 Amazon Simple Storage Service(Amazon S3) 버킷에 문서를 업로드합니다. 두 번째 S3 버킷은 기본 버킷에 대한 백업으로 사용됩니다. 개발자가 데이터 버킷에 파일을 업로드할 수 있어야 하지만 백업 버킷에 파일을 실수로 업로드할 수 없도록 해야 합니다. IAM 역할, S3 버킷 정책 및 IAM 권한 경계 조합을 사용하여 개발자에게 직무 수행에 필요한 최소 액세스 권한이 있도록 할 수 있습니다.

과제 1: EC2 인스턴스에 연결된 IAM 역할 보기 및 평가

이 과제에서는 EC2 인스턴스에 연결된 IAM 역할을 검토하여 수행할 액세스 권한이 있는 AWS 작업을 확인합니다.

  1. AWS 관리 콘솔 상단의 검색 창에서

    Copied
    를 검색하여 선택합니다.

  2. 페이지 왼쪽에 있는 탐색 창의 Instances에서 Instances를 선택합니다.

  3. Command Host 인스턴스 옆의 체크 박스를 선택합니다.

  4. 창 아래쪽의 Details 창에서 Security 탭을 선택합니다.

  5. Security details에서 IAM Role을 찾은 후 CommandHostIamRole 링크를 선택하여 새 웹 브라우저 탭에 표시되는 역할 세부 정보를 확인합니다.

  6. CommandHostIamRole 세부 정보 페이지의 Permissions policies 섹션에서 S3AccessPolicy 정책을 확장합니다.

파일 내용: S3AccessPolicy IAM 정책은 다음과 비슷합니다.

  • Resource 섹션에 있는 Amazon Resource Number(ARN)의 버킷 이름은 이러한 지침 왼쪽에 표시되는 DataBucketName 및 BackupBucketName의 값과 일치해야 합니다.
Copied 
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Action": [
        "s3:ListAllMyBuckets",
        "s3:ListBucket"
      ],
      "Resource": "*",
      "Effect": "Allow",
      "Sid": "AllowS3Read"
    },
    {
      "Action": [
        "s3:PutObject"
      ],
      "Resource": [
        "arn:aws:s3:::data-bucket-us-west-2-556254614/file1.txt",
        "arn:aws:s3:::data-bucket-us-west-2-556254614/file2.txt",
        "arn:aws:s3:::data-bucket-us-west-2-556254614/file3.txt",
        "arn:aws:s3:::data-bucket-us-west-2-556254614/file4.txt",
        "arn:aws:s3:::backup-bucket-us-west-2-556254614/file1.txt",
        "arn:aws:s3:::backup-bucket-us-west-2-556254614/file2.txt",
        "arn:aws:s3:::backup-bucket-us-west-2-556254614/file3.txt",
        "arn:aws:s3:::backup-bucket-us-west-2-556254614/file4.txt"
      ],
      "Effect": "Allow",
      "Sid": "AllowS3PutObjectToLabBuckets"
    }
  ]
}

고려 사항: 잠시 시간을 내어 정책을 검토합니다. 이 정책에서는 어떤 작업이 허용됩니까? 이러한 작업이 적용되는 특정 리소스가 있습니까?

이 정책은 EC2 인스턴스에 연결된 인스턴스 프로파일에 연결됩니다. 이 정책에는 다음과 같은 고유한 두 부분이 있습니다.

  • 첫 번째 부분에서는 EC2 인스턴스가 모든 버킷과 해당 버킷의 내용을 나열할 수 있습니다.
    • 버킷은 사용자 계정에 포함된 버킷, 공개적으로 액세스 가능한 버킷 또는 사용자가 액세스 권한을 부여한 다른 계정의 버킷일 수 있습니다.
  • 두 번째 부분에서 EC2 인스턴스는 객체를 data-bucket 및 backup-bucket에 업로드(PutObject)할 수 있지만 file1.txt, file2.txt, file3.txt 또는 file4.txt로 객체 이름이 명명 되어야 합니다.
  1. IAM 콘솔이 표시되는 웹 브라우저 탭을 닫고 EC2 콘솔이 있는 웹 브라우저 탭으로 돌아갑니다.

축하합니다! 이 실습 전체에서 작업하는 EC2 인스턴스에 연결된 IAM 역할 및 정책을 검토했습니다.

과제 2: EC2 인스턴스에 부여된 권한 유효성 검사

과제 2.1: 명령 호스트 인스턴스에 연결

먼저, AWS Systems Manager Session Manager를 사용하여 Amazon EC2 인스턴스에 연결합니다.

  1. 목록에서 이 지침의 왼쪽에 있는 CommandHostSessionUrl 값을 복사한 다음, 새 웹 브라우저 탭에 붙여넣습니다.

예상 출력: 콘솔은 웹 브라우저 창 내부의 인스턴스에 연결됩니다. 인스턴스에 연결하면 일련의 명령이 자동으로 실행되어 사용자의 홈 디렉터리로 변경하고, 다양한 환경 변수를 설정하고, 다음과 같이 작업 디렉터리의 경로를 표시합니다.

과제 2.3: EC2 인스턴스에 부여된 권한 테스트

다음으로, 인스턴스 프로파일과 역할을 통해 EC2 인스턴스에 연결된 S3AccessPolicy라는 IAM 정책을 테스트합니다. 인스턴스에서 AWS CLI 명령을 사용하여 승인된 작업 및 승인되지 않은 작업을 수행하고 결과를 기록해 둡니다.

실습 환경 구축 프로세스의 일부로, Session Manager를 통해 EC2 인스턴스에 연결할 때의 기본 설정을 정의하는 AWS Systems Manager 문서 SSM-SessionManagerRunShell이 생성되었습니다. 해당 문서에서는 인스턴스에 연결할 때 명령을 실행할 수 있습니다. 이 실습에서는 명령을 더 쉽게 입력하도록 하기 위해 상호 작용하는 두 S3 버킷의 이름을 저장하는 환경 변수를 생성하는 명령이 실행됩니다. 자세한 내용은 이 실습의 마지막에 있는 추가 리소스 섹션의 AWS Systems Manager Session Manager를 참조하십시오.

S3AccessPolicy 정책 문서의 첫 번째 스테이트먼트에서는 해당 인스턴스에 대해 특정 read 작업을 부여합니다. 첫 번째 작업 s3:ListAllMyBuckets에서는 인스턴스가 계정과 연결된 모든 버킷의 이름을 나열할 수 있습니다. 두 번째 작업인 s3:ListBucket에서는 인스턴스가 지정된 버킷에 있는 모든 객체의 이름을 나열할 수 있습니다. 다음은 예상되는 해당 스테이트먼트가 어떤 형태인지를 생각나게 하는 내용입니다.

Copied 
...
{
  "Action": [
    "s3:ListAllMyBuckets",
    "s3:ListBucket"
  ],
  "Resource": "*",
  "Effect": "Allow"
},
...
  1. 다음 명령을 실행하여 계정과 연결된 모든 버킷을 나열합니다(s3:ListAllMyBuckets 권한).
Copied 
aws s3 ls

예상 출력: 출력은 이러한 명령 왼쪽에 나열된 DataBucketName 및 BackupBucketName 값과 일치하는 이름을 갖는 2개 이상의 버킷을 표시합니다.

  1. 다음 명령을 실행하여 data-bucket 버킷에 나열된 모든 객체를 나열합니다(s3:ListBucket 권한).
Copied 
aws s3 ls $DataBucket

예상 출력: 버킷이 현재 비어 있으므로 출력이 없습니다.

다음으로 s3:PutObject 권한을 테스트합니다. 정책 문서는 리소스 ARN을 사용하여 명확하게 명명된 파일(file1.txt, file2.txt, file3.txt 또는 file4.txt) 및 사용자 계정의 버킷에 대해서만 작업을 수행할 수 있도록 합니다. 나열된 4개의 이름을 갖는 파일을 data-bucket 및 backup-bucket에 업로드할 수 있습니다. 다음은 예상되는 해당 스테이트먼트가 어떤 형태인지를 생각나게 하는 내용입니다.

Copied 
...
{
  "Action": [
    "s3:PutObject"
  ],
  "Resource": [
    "arn:aws:s3:::data-bucket-us-west-2-556254614/file1.txt",
    "arn:aws:s3:::data-bucket-us-west-2-556254614/file2.txt",
    "arn:aws:s3:::data-bucket-us-west-2-556254614/file3.txt",
    "arn:aws:s3:::data-bucket-us-west-2-556254614/file4.txt",
    "arn:aws:s3:::backup-bucket-us-west-2-556254614/file1.txt",
    "arn:aws:s3:::backup-bucket-us-west-2-556254614/file2.txt",
    "arn:aws:s3:::backup-bucket-us-west-2-556254614/file3.txt",
    "arn:aws:s3:::backup-bucket-us-west-2-556254614/file4.txt"
  ],
  "Effect": "Allow",
  "Sid": "AllowS3PutObjectToLabBuckets"
}
...
  1. 다음 명령을 실행하여 file1.txt를 data-bucket으로 업로드합니다.
Copied 
aws s3 cp /home/ec2-user/lab1/file1.txt s3://$DataBucket

과제 3: 리소스 기반 정책을 S3 버킷에 적용하여 파일 업로드 거부

이 과제에서는 IAM 정책을 만들어 S3 버킷에 연결합니다. 이것은 리소스 기반 정책의 예에 해당합니다. 그런 다음, AWS CLI 명령을 사용하여 파일 업로드를 다시 시도합니다(s3:PutObject 작업). 이전에는 이 작업이 두 버킷 모두에 허가되었지만 리소스 기반 정책을 적용한 후에는 backup-bucket에 대해 명시적으로 거부됩니다.

리소스 기반 정책을 모든 AWS 리소스 유형에 사용할 수 있는 것은 아닙니다. 지원되는 리소스에 대한 내용은 이 실습 마지막에 나오는 추가 리소스 섹션의 IAM을 사용한 AWS 서비스를 참조하십시오.

과제 3.1: S3 버킷 정책 생성

  1. S3 콘솔이 있는 웹 브라우저 탭으로 돌아갑니다.

  2. 페이지 왼쪽의 탐색 창에서 Buckets를 선택합니다.

  3. Buckets 페이지에서 backup-bucket으로 시작하는 버킷 이름의 링크를 선택합니다.

  4. bucket-bucket 세부 정보 페이지에서 Permissions 탭을 선택합니다.

  5. Bucket policy 섹션에서 Edit을 선택합니다.

  6. Edit bucket policy 페이지의 Policy 섹션에서 아래 정책 블록을 추가합니다. 이 때, 아래 내용을 꼭 참조하십시오.

  • INSERT_COMMAND_HOST_IAM_ROLE_ARN 플레이스 홀더 값을 이러한 지침 왼쪽에 나열된 CommandHostIamRoleArn 값으로 바꿉니다(따옴표를 유지해야 함).
  • INSERT_BACKUP_BUCKET_NAME 플레이스 홀더 값을 이러한 지침 왼쪽에 나열된 BackupBucketName 값으로 바꿉니다.
Copied 
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Sid": "DenyPutObjectBackupBucket",
      "Effect": "Deny",
      "Principal": {
        "AWS": "INSERT_COMMAND_HOST_IAM_ROLE_ARN"
      },
      "Action": "s3:PutObject",
      "Resource": "arn:aws:s3:::INSERT_BACKUP_BUCKET_NAME/*"
    }
  ]
}

참고: 정책은 Principal 파라미터에 지정된 IAM 역할과 연결된 모든 리소스로 파일(s3:PutObject)을 업로드하기 위한 액세스 권한을 거부합니다. 이 경우 역할이 EC2 인스턴스에 연결된 인스턴스 프로파일에 연결됩니다.

  1. 페이지 하단에서 Save changes를 선택합니다.

과제 3.2: 버킷 정책의 효과 검증

지금까지 backup-bucket 버킷에 버킷 정책을 적용했으므로 자격 증명 기반(IAM 역할)과 리소스 기반(버킷 정책)이 모두 적용된 후 그로 인한 EC2 인스턴스 권한을 테스트하고 기록합니다. EC2 인스턴스는 s3:PutObject 작업을 허용하는 인스턴스 프로파일을 통해 IAM 역할에 할당됩니다. 그렇지만 backup-bucket에는 이 작업을 명시적으로 거부하는 리소스 기반 정책이 할당됩니다. 최종 결과는 backup-bucket이 아닌 data-bucket에 파일을 업로드하기 위한 권한이어야 합니다.

  1. 명령 호스트 인스턴스에 콘솔이 연결된 웹 브라우저 탭으로 돌아갑니다.

  2. 다음 명령을 실행하여 file2.txt를 data-bucket으로 업로드합니다.

Copied 
aws s3 cp file2.txt s3://$DataBucket

예상 출력: 출력에는 다음과 비슷하게 파일이 업로드되었음을 나타내는 업로드 메시지가 표시됩니다.

Copied 
upload: ./file2.txt to s3://data-bucket-us-west-2-556254614/file2.txt
  1. 다음 명령을 통해 data-bucket의 내용을 나열하여 파일이 업로드되었음을 확인합니다.
Copied 
aws s3 ls $DataBucket

예상 출력: 출력은 다음과 비슷하게 2개의 파일을 표시해야 합니다.

Copied 
2022-12-07 19:39:05         61 file1.txt
2022-12-07 19:47:44         61 file2.txt

file2.txt가 data-bucket에 업로드되었음을 확인합니다. 다음으로 동일한 파일을 backup-bucket에 업로드합니다.

  1. 다음 명령을 실행하여 file2.txt를 backup-bucket으로 업로드합니다.
Copied 
aws s3 cp file2.txt s3://$BackupBucket

예상 출력: 출력은 다음과 비슷하게 upload failed 및 AccessDenied 메시지를 표시해야 합니다.

Copied

  1. 다음 명령을 실행하여 backup-bucket의 내용을 나열합니다.
Copied 
aws s3 ls $BackupBucket

예상 출력: 출력은 다음과 비슷하게 1개의 파일을 표시해야 합니다.

Copied 
2022-12-07 19:39:41         61 file1.txt

참고: file2.txt를 data-bucket에 업로드하기 위한 첫 번째 명령이 성공적으로 수행되었지만 backup-bucket에 대한 업로드가 실패했습니다. EC2 인스턴스 프로파일을 사용하면 모든 버킷으로 파일을 업로드할 수 있지만(s3:PutObject) backup-bucket에 대한 버킷 정책은 파일 업로드를 거부합니다. 최종 결과는 인스턴스에서 backup-bucket이 아닌 data-bucket으로 업로드할 수 있다는 것입니다.

축하합니다! 리소스 기반 정책(S3 버킷 정책)이 의도한 대로 작동하는지 확인했습니다.

과제 4: IAM 권한 경계 적용

일부 경우에 할당된 정책과 관계없이 IAM 사용자 또는 역할에 허용되는 최대 권한을 정의하려고 할 수 있습니다. 권한 경계는 관리형 정책을 사용하여 자격 증명 기반 정책을 통해 설정할 수 있는 최대 권한을 IAM 엔터티에 부여하는 고급 기능입니다. 엔터티의 권한 경계에 따라 엔터티는 자격 증명 기반 정책 및 관련 권한 경계 모두에서 허용하는 작업만 수행하도록 허용합니다.

이 과제에서 명령 호스트 EC2 인스턴스에 연결된 IAM 역할에 권한 경계를 추가하여 data-bucket 버킷으로 업로드할 수 있는 파일을 추가로 제한합니다.

과제 4.1: IAM 권한 경계 생성

  1. S3 콘솔이 있는 웹 브라우저 탭으로 돌아갑니다.

  2. AWS 관리 콘솔 상단의 검색 창에서

    Copied
    을 검색하여 선택합니다.

  3. 페이지 왼쪽 탐색 창의 Access Management에서 Roles를 선택합니다.

  4. Roles 페이지에서 CommandHostIamRole 역할에 대한 링크를 선택하여 세부 정보를 봅니다.

  5. CommandHostIamRole 페이지의 Permissions boundary 섹션에서 Set permissions boundary를 선택합니다.

  6. Set permissions boundary 페이지에서 S3PermissionsBoundary를 선택합니다.

파일 내용: 실습 환경 구축 프로세스 동안 S3PermissionsBoundary IAM 정책이 생성되었습니다. 과제 1에서 검토한 IAM 정책에 대해 비슷한 권한이 있지만 객체 이름이 file3.txt인 경우 data-bucket에 대한 업로드만 허용합니다. backup-bucket에 대한 업로드는 허용하지 않습니다. 정책은 다음과 유사합니다.

Copied 
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Action": [
        "s3:ListAllMyBuckets",
        "s3:ListBucket"
      ],
      "Resource": "*",
      "Effect": "Allow"
    },
    {
      "Action": [
        "s3:PutObject"
      ],
      "Resource": [
        "arn:aws:s3:::data-bucket-us-west-2-556254614/file3.txt"
      ],
      "Effect": "Allow"
    }
  ]
}

참고: 또한 이 정책을 사용하면 인스턴스 콘솔에 연결할 수 있도록 통해 AmazonSSMManagedInstanceCore AWS 관리형 정책에 있는 동일한 작업을 수행할 수 있습니다.

  1. 페이지 하단에서 Set Boundary를 선택합니다.

예상 결과: Successfully set permissions boundary for CommandHostIamRole 메시지가 있는 녹색 배너가 페이지 상단에 표시됩니다.

IAM 권한 경계에 대한 자세한 내용은 이 실습의 마지막에 있는 추가 리소스 섹션에서 IAM 엔터티에 대한 권한 경계 및 IAM 권한 경계를 사용하는 경우 및 위치를 참조하십시오.
