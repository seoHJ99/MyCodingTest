실습 개요

이 실습에서는 Linux 기반 Amazon Elastic Compute Cloud(Amazon EC2) 인스턴스에 Amazon CloudWatch Logs 에이전트를 설치합니다. 그런 다음 에이전트가 CloudWatch Logs로 시스템 로그를 보내 인스턴스에서 실패한 인증 시도를 모니터링하도록 구성합니다. 마지막으로 VPC 흐름 로그가 네트워크 트래픽을 캡처하도록 구성한 다음 CloudWatch Logs Insight를 사용하여 로그를 분석합니다.

목표

본 실습을 마치면 다음을 할 수 있습니다.

  • Linux 기반 EC2 인스턴스에 Amazon CloudWatch Logs 에이전트를 설치합니다.
  • 시스템 인증 로그를 Amazon CloudWatch 로그로 보냅니다.
  • VPC 흐름 로그를 생성하여 네트워크 트래픽을 캡처합니다.
  • CloudWatch Logs Insights를 사용하여 로그를 분석합니다.

기본적으로 CloudWatch 에이전트는 루트 사용자로 실행됩니다. 애플리케이션 서비스와 시스템 서비스에는 항상 전용 사용자를 사용하는 것이 모범 사례입니다. CloudWatch 에이전트 설치 관리자는 cwagent라는 새 사용자를 생성하여 사용자가 사용하기를 선택하거나 다른 사용자를 생성할 수 있게 합니다. 어떤 경우에도 모두 CloudWatch 에이전트를 실행하도록 구성한 사용자는 CloudWatch로 보낼 로그 파일에 대한 읽기 및 실행 액세스 권한이 있어야 합니다. 자세한 정보는 이 실습 마지막의 추가 리소스 섹션에서 CloudWatch 에이전트를 다른 사용자로 실행을 참조하십시오.

  1. 다음 명령을 실행하여 cwagent 사용자에게 /var/log/secure 파일에 대한 읽기 및 실행 액세스 권한을 부여합니다.

  1. Jump Host 인스턴스에 콘솔이 연결된 웹 브라우저 탭으로 돌아갑니다.

  2. 다음 명령을 실행하여 Database Server 인스턴스에 대한 SSH 연결을 student 사용자로 설정합니다.

  • DB_SERVER_IP 자리 표시자 값을 이 지침의 왼쪽에 나열된 DbServerIp 값으로 바꿉니다.
Copied 
ssh student@DB_SERVER_IP

  1. 비밀번호를 입력하라는 메시지가 나타나면 잘못된 비밀번호를 세 번 입력합니다. 인증 실패를 생성하려는 것이므로, 지금은 로그인에 실패해야 합니다.

  2. 앞의 두 단계를 두 차례 더 반복하여 총 3번의 로그인 실패를 생성합니다.

  3. VPC 관리 콘솔이 있는 웹 브라우저 탭으로 돌아갑니다.

  4. 페이지 상단의 통합 검색 창에서

    Copied
    를 검색하여 선택합니다.

  5. 페이지 왼쪽에 있는 탐색 창의 Logs 섹션에서 Log groups를 선택합니다.

  6. Log groups 페이지에서 lab_vpc_flow_log에 대한 링크를 선택합니다.

  7. lab_vpc_flow_log 세부 정보 페이지의 Log streams 섹션에서 Search all log streams를 선택합니다.

Search all log streams 옵션을 사용하면 개별적으로 검색하지 않고 로그 그룹에 있는 모든 로그 스트림을 검색할 수 있습니다.

  1. 몇 가지 로그 항목을 검토하여 허용된 트래픽 유형과 거부된 트래픽 유형을 확인합니다. 오른쪽 화살표()를 사용하여 각 항목을 확장하여 전체 메시지를 볼 수 있습니다.

어떤 유형의 트래픽이 보입니까? 이전 샘플 로그 데이터에는 다음과 같은 항목이 있습니다

  • 10.10.1.x IP 범위는 Jump Host 인스턴스가 있는 실습 퍼블릭 서브넷입니다.
  • 10.10.10.x IP 범위는 Database Server 인스턴스가 있는 실습 프라이빗 서브넷입니다.
  • 포트 443의 트래픽과 퍼블릭 IP 주소와 10.10.1.x IP 범위 사이의 임의 포트 번호는 Jump Host 인스턴스에 대한 세션 관리자 연결에 의해 생성됩니다.
  • 포트 22의 트래픽과 10.10.1.x와 10.10.10.x IP 범위 사이의 임의 포트 번호는 Jump Host 인스턴스에서 Database Server 인스턴스로의 SSH 연결에 의해 생성됩니다.

축하합니다! VPC 내의 네트워크 트래픽을 모니터링하기 위해 VPC 흐름 로그를 생성하는 데 성공했습니다.

태스크 3.4: CloudWatch Logs Insight를 사용하여 SSH 연결 시도 및 인증 실패 검색

로그를 통해 해당 이벤트를 수동으로 검색하는 대신 Amazon CloudWatch Logs Insight를 사용하여 여러 로그 그룹의 데이터를 동시에 쿼리할 수 있습니다. 이 시나리오에서는 데이터베이스 서버 보안 로그와 VPC 흐름 로그를 검색하여 실패한 인증 시도를 자세히 살펴보려 합니다.

  1. 페이지 왼쪽에 있는 탐색 창의 Logs에서 Log Insights를 선택합니다.

  2. Select log group(s) 드롭다운 메뉴에서 database_server_security_logs와 lab_vpc_flow_log를 선택합니다.

  3. 쿼리 상자에서 기존 쿼리를 삭제하고 다음 쿼리를 입력하여 두 로그에서 포트 22(SSH)의 VPC 트래픽과 Database Server 인스턴스의 인증 실패를 검색합니다.

Copied 
fields @timestamp, @message
| filter (@message like / 22 / or @message like /authentication failure/)
| sort @timestamp desc
| limit 25
  1. Run query를 선택합니다.

예상 출력: 쿼리는 두 로그 모두에서 결과를 표시해야 하며, 일부는 (데이터베이스 서버 보안 로그에서) authentication failure 메시지를 포함하고 나머지는 로그와 실습 VPC 내의 SSH 트래픽 흐름을 보여주는 로그를 포함해야 합니다.

참고: VPC 흐름 로그의 쿼리에 항목이 없으면 2~3분 기다린 후 쿼리를 다시 실행합니다.

고려 사항: SSH 연결 시도 및 인증 실패 원인에 대한 추가 정보를 얻기 위해 다양한 로그에서 타임스탬프를 일치시킬 수 있습니까?

CloudWatch Logs Insights를 사용한 로그 데이터에 대한 자세한 내용은 이 실습 마지막에 있는 추가 리소스 섹션의 CloudWatch Logs Insights 쿼리 구문을 참조하십시오.

축하합니다! VPC 흐름 로그를 만들고 여러 로그의 데이터를 분석했습니다.
