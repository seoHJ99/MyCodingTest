태스크 1: WAF Automation on AWS CloudFormation 템플릿이 생성한 리소스를 검토합니다.

WAF Automation on AWS CloudFormation 스택은 실습 환경 구축 프로세스 중에 구축되었습니다. 이 태스크에서는 해당 스택이 생성한 리소스를 검토합니다.

AWS에서 WAF 자동화 솔루션은 일반적인 웹 기반 공격을 필터링하는 일련의 AWS WAF 규칙을 자동 배포합니다. AWS WAF 웹 액세스 제어 목록(웹 ACL)에 포함된 규칙을 정의하는 미리 구성된 보호 기능 중에서 선택할 수 있습니다. 솔루션 배포 후 AWS WAF는 웹 요청을 검사하여 Amazon CloudFront 배포 또는 Application Load Balancer를 보호할 수 있습니다.

AWS WAF를 사용하여 공격 패턴을 차단하는 사용자 지정 애플리케이션별 규칙을 생성하여 애플리케이션 가용성을 보장하고 리소스를 보호하며 과도한 리소스 사용을 방지할 수 있습니다.

WAF Automation on AWS 솔루션은 최신 버전의 AWS WAF(AWS WAFV2) 서비스 API를 지원합니다.

자세한 내용은 추가 리소스 섹션의 WAF Automation on AWS를 참조하십시오.

태스크 1.1: CloudFormation의 리소스 목록 검토

CloudFormation 템플릿을 배포할 때 템플릿이 생성한 모든 리소스의 목록을 볼 수 있습니다.

  1. AWS 관리 콘솔 상단의 검색 창에서

    Copied
    을 검색하여 선택합니다.

  2. AWS CloudFormation 페이지의 왼쪽 탐색 창에서 Stacks을 선택합니다.

참고: Stacks 페이지에서 두 개의 스택(하나는 이름에 WafAutomation이 있고 다른 하나는 이름에 WebACLStack이 있음)이 습니다.

  1. 이름에 WafAutomation이 있는 스택의 링크를 선택합니다.

  2. 스택 세부 정보 페이지에서 Resources 탭을 선택합니다.

리소스 탭에는 템플릿이 생성한 모든 AWS 리소스가 나열됩니다. 목록의 일부 항목에는 생성된 리소스로 쉽게 이동할 수 있는 링크가 포함되어 있습니다.

  1. 리소스 목록을 검토하여 이 템플릿이 만든 내용을 파악합니다.

IAM 역할, Lambda 함수 및 CloudWatch 대시보드와 같은 리소스를 찾아야 합니다.

다음으로 보조 스택이 생성한 AWS WAF 웹 ACL 관련 리소스를 검토합니다.

  1. Stacks 창에서 이름에 WebACLStack이 있는 스택을 선택합니다.

  2. Resources 탭을 선택합니다.

  3. 리소스 목록을 검토하여 이 템플릿이 만든 내용을 파악합니다.

AWS WAF 웹 ACL, IP 집합, 사용자 지정 타이머 및 Lambda 함수와 같은 리소스를 찾아야 합니다.

태스크 1.2: CloudFormation 템플릿이 생성한 WAF 리소스 검토

WAF Automation on AWS CloudFormation 템플릿은 여러 리소스를 생성하지만 이 태스크에서는 이 실습과 관련된 리소스에 집중합니다.

  1. AWS 관리 콘솔 상단의 검색 창에서

    Copied
    를 검색하여 선택합니다.

  2. 페이지 왼쪽의 WAF & Shield 탐색 창에서 AWS WAF 아래의 Web ACLs를 선택합니다.

  3. Web ACLs 페이지에 US East (N. Virginia) 로 기본 설정된 AWS 리전 버튼이 있습니다. 이름에 WafAutomation이 있는 웹 ACL이 나열되지 않은 경우 AWS 리전 드롭다운 버튼을 선택하여 이 지침의 왼쪽에 나열된 AwsRegionName 값으로 변경합니다.

  4. 이름에 WafAutomation이 있는 링크를 선택하여 세부 정보를 확인합니다.

  5. 웹 ACL 세부 정보 페이지에서 Rules 탭을 선택합니다.

  6. Rules 섹션에는 WAF Automation on AWS CloudFormation 템플릿을 배포할 때 선택한 옵션과 부합하는 규칙이 나열되어 있습니다.

다음과 비슷한 규칙이 표시됩니다.

이름 작업 우선순위 사용자 지정 응답
AWS-AWSManagedRulesCommonRuleSet 규칙 작업 사용 0 -
lab-WafAutomation-XWMGEMK936DKWhitelistRule 허용 1 -
lab-WafAutomation-XWMGEMK936DKBlacklistRule 차단 2 -
lab-WafAutomation-XWMGEMK936DKHttpFloodRateBasedRule 차단 4 -

참고: 규칙 이름은 AWS CloudFormation 스택 이름으로 결정됩니다. 이 실습에서는 WAF Automation on AWS 템플릿을 중첩된 스택으로 배포하므로 이름이 환경에 있을 수 있는 이름보다 깁니다.

이 실습에서는 HTTP 플러드 속도 기반 규칙 및 SQL 명령어 삽입 규칙에 대해 테스트합니다. 다음에는 이 두 규칙의 세부 정보를 검토합니다.

  1. HttpFloodRateBasedRule 문구가 포함된 규칙 링크를 선택합니다.

  2. HttpFloodRateBasedRule 페이지에서 Rule 및 Action 세부 정보를 검토합니다.

실습 환경 구축 프로세스 중에 CloudFormation 템플릿을 배포할 때 HTTP 플러드 속도 제한의 기본값은 5분 내에 100개의 연결로 설정되었습니다. HttpFloodRateBasedRule 규칙은 이 규칙이 속한 웹 ACL에 할당된 리소스에 대한 HTTP 연결 시도가 100회를 초과하는 경우 연결을 시도하는 원본 IP를 차단하도록 지정합니다.

  1. 웹 브라우저의 뒤로 버튼을 사용하여 웹 ACL 세부 정보 페이지로 돌아갑니다.

  2. Rules 탭을 선택합니다.

  3. Rules 섹션에서 SqlInjectionRule 문구가 포함된 규칙 링크를 선택합니다.

  4. SqlInjectionRule 페이지에서 규칙에 포함된 다양한 스테이트먼트와 작업을 검토합니다.

SqlInjectionRule 규칙은 쿼리 문자열 또는 URI 경로와 같은 여러 쿼리 유형에서 악의적인 SQL 코드를 찾습니다. 나열된 양식 중 하나에서 SQL 명령어 삽입 공격이 탐지되면 요청이 차단됩니다.

  1. 웹 브라우저의 뒤로 버튼을 사용하여 웹 ACL 세부 정보 페이지로 돌아갑니다.

축하합니다! WAF Automation on AWS CloudFormation 템플릿을 성공적으로 배포하고 이를 통해 생성된 웹 ACL 및 규칙을 검증했습니다.

태스크 2: HTTP 플러드 방지 검토

이 태스크에서는 손상된 호스트의 HTTP 플러드 서비스 거부 공격을 시뮬레이션합니다. Amazon Linux 2 EC2 인스턴스가 별도의 VPC 및 서브넷에 배포되어 네트워크 외부의 엔드포인트에서 발생하는 공격을 시뮬레이션했습니다. AWS Systems Manager Session Manager를 사용하여 공격자 인스턴스에 연결합니다.

먼저 WAF 보호가 없는 상태에서 공격을 시뮬레이션합니다. 그런 다음 애플리케이션 로드 밸런서를 AWS WAF 웹 ACL의 리소스로 추가하고 공격을 다시 시뮬레이션하여 보호 기능의 실행을 관찰합니다.

태스크 2.1: HTTP 요청으로 웹 사이트 플러딩 시도

  1. 목록에서 이 지침의 왼쪽으로 InstanceSessionUrl 값을 복사한 다음, 새 웹 브라우저 탭에 붙여 넣습니다.

콘솔은 웹 브라우저 창 내부의 인스턴스에 연결됩니다. 인스턴스에 연결하면 일련의 명령이 자동으로 실행되어 사용자의 홈 디렉터리로 변경하고 다음과 같이 작업 디렉터리의 경로를 표시합니다.

  1. 터미널 세션에서 다음 명령을 실행하여 HTTP를 애플리케이션 로드 밸런서에 10초마다 100회 연결하고 3분 동안 실행하는 스크립트를 만듭니다.
Copied 
cat <<EOF > loop.sh
#!/bin/bash

runtime="3 minute"
endtime=\$(date -ud "\$runtime" +%s)
totaltime=0

while [[ \$(date -u +%s) -le \$endtime ]]
do
    curl -I "$AlbUrl/?[1-100]" | grep HTTP
    echo "sleeping for 10 seconds"
    echo "total time running = \$totaltime seconds"
    ((totaltime=totaltime+10))
    sleep 10
done
EOF

예상 출력: 이 명령은 출력이 없습니다.

  1. 다음 명령을 실행하여 10초마다 100번씩 HTTP를 사용하는 애플리케이션 로드 밸런서를 통해 샘플 웹 사이트에 연결하도록 만들었던 스크립트를 활성화합니다.
Copied 
sh loop.sh

이 스크립트는 HTTP 요청이 차단된 시간 동안 총 3분 동안 실행됩니다. 애플리케이션 로드 밸런서를 보호하는 AWS WAF 웹 ACL 규칙이 없으면 웹 사이트 성능이 저하되거나 사용자가 수동으로 개입할 때까지 웹 사이트 액세스가 손실되어 로그에 기록될 때까지 이러한 유형의 공격이 계속될 수 있습니다.

태스크 2.2: 애플리케이션 로드 밸런서를 AWS WAF 웹 ACL에서 리소스로 추가

그런 다음 애플리케이션 로드 밸런서 리소스를 웹 ACL에 연결하여 HTTP 플러드 공격으로부터 보호합니다.

  1. AWS WAF 웹 ACL 세부 정보 페이지가 있는 웹 브라우저 탭으로 돌아갑니다. 좌측 페이지의 Web ACLs를 선택한 다음에 WafAutomation 문구가 포함된 웹 ACL 링크를 클릭합니다.

  2. 웹 ACL 세부 정보 페이지에서 Associated AWS resources 탭을 선택합니다.

  3. Associated AWS resources 섹션에서 Add AWS resources를 선택합니다.

  4. Add AWS resources 팝업 창에서

  • Resource type에 Application Load Balancer를 선택합니다.
  • Name에 web-server-alb를 선택합니다.

참고: web-server-alb 애플리케이션 로드 밸런서가 랩 환경 구축 프로세스 중에 생성되었습니다. 퍼블릭 서브넷에 배포되고 프라이빗 서브넷에 배포된 웹 서버의 프런트 엔드로 사용됩니다.

  1. Add를 선택합니다.

태스크 3: SQL 명령어 삽입 보호 검사
