실습 개요

이 실습에서는 불량 행위자에 의해 손상될 가능성이 있는 EC2 인스턴스에 대한 경고를 수신하는 사고 대응 팀의 구성원으로 활동합니다. 효과적인 조사, 분석 및 학습된 교훈을 위해 입증된 프로세스와 기술을 사용하여 인시던트에 대응해야 합니다.

목표

본 실습을 마치면 다음을 할 수 있습니다.

  • 손상된 인스턴스 메타데이터 및 영구 디스크를 캡처합니다.
  • 손상된 인스턴스의 스냅샷을 생성합니다.
  • 인스턴스를 격리하고 실수로 인스턴스가 종료되지 않도록 보호합니다.
  • 시스템 로그를 검토하여 위반으로 의심되는 사항을 확인합니다.
  • 인스턴스 설정을 업데이트하여 취약성을 완화합니다.
  • 향후 유사한 인시던트를 처리할 수 있도록 자동 인시던트 대응을 작성합니다.

필수 기술 지식

이 실습을 정상적으로 완료하려면 AWS Cloud Practitioner Essentials 교육 과정에서 정의한 AWS 관련 기본 지식을 갖춰야 합니다. AWS 관리 콘솔을 탐색하는 것과 기본 네트워킹 개념에 익숙해야 합니다.

과제 1: 모든 손상된 인스턴스 영구 디스크 및 메타데이터 캡처

이 과제에서는 인시던트를 검토하여 손상된 인스턴스 세부 정보를 식별합니다. 그런 다음 초기 대응 과제를 수행하여

  • 손상된 인스턴스 메타데이터를 캡처합니다.
  • 손상된 인스턴스에 대해 종료 방지 기능을 활성화합니다.
  • 인스턴스 메모리와 디스크를 캡처합니다.
  • 캡처된 스냅샷에서 새 인스턴스를 배포하여 해당 인스턴스를 자세히 조사합니다.
  • 손상된 인스턴스에 태그를 지정하고 해제하고 격리합니다.

과제 1.1: 손상된 인스턴스 세부 정보 검토

잠재적으로 손상된 EC2 인스턴스에 대한 인시던트 알림을 받는 인시던트 대응 팀 구성원인 여러분은 손상된 인스턴스 세부 정보를 검토하고 잠재적 인시던트에 대한 정보와 아티팩트를 수집하여 인시던트 대응 작업을 시작할 수 있어야 합니다.

  1. 이 지침의 왼쪽에 제공된 리소스를 확인하고 인시던트 결과에 보고된 다음 세부 정보를 기록합니다.
  • CompromisedInstanceId
  • CompromisedInstanceDNSName

실습에서 손상된 인스턴스의 이름 태그는 compromised-ec2입니다.

과제 1.2: 손상된 인스턴스 메타데이터 캡처

이제 메타데이터를 수집하여 손상된 인스턴스 정보를 수집합니다.

  1. jump-server 터미널에 연결하려면 이 지침 왼쪽에 있는 목록에서 SessionManagerJumpServerUrl 값을 복사하여 새 브라우저 탭에서 이 URL을 엽니다. 이렇게 하면 jump-server$ 셸 프롬프트와 함께 jump-server 터미널이 열립니다.

  2. 명령: jump-server 터미널에서 아래 명령 블록을 입력합니다. 이를 통해 변수를 재사용할 수 있도록 손상된 인스턴스 ID, 리전, IR 버킷 및 메타데이터 파일을 터미널 셸 변수로 설정합니다. 이 변수를 설정하려면 값을 목록에서 해당 지침의 왼쪽으로 옮긴 후 다음 명령을 실행합니다.

명령 복사: 이 명령을 텍스트 편집기에 복사하고 값을 실습 지침의 왼쪽 패널을 참고하여 바꾼 다음 터미널에 다시 붙여넣기 합니다.

jump-server$

Copied 
compromised_instance_id=REPLACE_WITH_YOUR_CompromisedInstanceId
lab_region=REPLACE_WITH_YOUR_LabRegionCode
ir_bucket=REPLACE_WITH_YOUR_IRBucketName
metadata_file=metadata_$compromised_instance_id.json

예상 출력:

오류가 발생하지 않는 한 없습니다.

  1. 명령: 변수가 올바르게 설정되었는지 확인하기 위해 다음 명령을 실행하고 출력을 확인합니다.

jump-server$

Copied 
echo $compromised_instance_id
echo $lab_region
echo $ir_bucket
echo $metadata_file

예상 출력:

Copied 
******************************
**** This is OUTPUT ONLY. ****
******************************

jump-server$ echo $compromised_instance_id
i-0cf62dd022866dce4
jump-server$ echo $lab_region
us-east-1
jump-server$ echo $ir_bucket
irbucket-random_number
metadata_i-0cf62dd022866dce4.json
  1. 명령: 인스턴스 메타데이터를 캡처하여 파일에 저장하려면 다음 명령을 실행합니다.

jump-server$

Copied 
aws ec2 describe-instances --instance-ids $compromised_instance_id --region $lab_region >> $metadata_file

예상 출력:

오류가 발생하지 않는 한 없습니다.

  1. 인스턴스 메타데이터가 캡처되었는지 확인하려면 위에서 만든 파일의 내용을 확인하고 다음 명령을 실행합니다.

jump-server$

Copied 
cat $metadata_file

예상 출력: 손상된 인스턴스 세부 정보가 모두 표시됩니다.

  1. 인시던트 대응 아티팩트 수집의 일부로 메타데이터 파일을 IR 버킷에 업로드하려면 다음 명령을 실행합니다.

jump-server$

Copied 
aws s3api put-object --bucket $ir_bucket --body $metadata_file --key $metadata_file

예상 출력:

Copied 
******************************
**** This is OUTPUT ONLY. ****
******************************

    "ETag": "\"68b329da9893e34099c7d8ad5cb9c940\""
}

  1. IR 버킷의 메타데이터 파일을 확인하려면 AWS 관리 콘솔 상단의 검색 창에서

    Copied
    를 검색하여 선택합니다.

  2. S3 Management Console의 버킷에서 버킷이 나열된 위치를 찾고 자신의 버킷에 대한 ir-bucket-<RANDOM_NUMBER> 링크를 선택합니다.

  3. 버킷에서 Objects 탭을 선택하면 버킷에 업로드된 메타데이터 파일이 표시됩니다.

참고: 객체를 다운로드하고 내용을 확인할 수 있습니다.

과제 1.3: 손상된 인스턴스에서 종료 방지 기능 활성화

손상된 인스턴스의 스냅샷을 만들더라도 인스턴스 세부 정보와 디스크를 캡처하는 동안 실수로 종료되지 않도록 보호해야 합니다. 모든 데이터 수집이 완료될 때까지 인스턴스를 보호하려면 종료 방지 기능을 활성화해야 합니다.

  1. AWS 관리 콘솔 상단의 검색 창에서

    Copied
    를 검색하여 선택합니다.

  2. 왼쪽 탐색 창에서 Instances 섹션에서 Instances를 선택합니다.

  3. Instances에서 compromised-ec2 인스턴스를 찾아 이름 옆에 있는 를 선택합니다.

  4. 페이지 오른쪽 상단에서 Actions 를 선택한 다음 Instance settings를 선택하고 Change termination protection을 선택합니다.

  5. Change termination protection 팝업 창에서 Enable을 선택한 다음 Save를 선택합니다.

예상 출력:

Successfully enabled termination protection for instance

손상된 인스턴스에서 종료 방지 기능을 활성화했습니다. 이 기능을 해제할 때까지 종료할 수 없습니다.

  • 참고: 인시던트 대응의 일환으로, 첨부된 모든 볼륨에 대해 ‘DeleteOnTermination’ 설정을 비활성화할 수도 있습니다. 이렇게 하면 인스턴스가 종료된 경우 볼륨이 보존됩니다. 이 실습에서는 이 옵션을 설정하지 않습니다. 이 기능에 대한 자세한 내용은 여기에서 확인할 수 있습니다. Disable the “DeleteOnTermination” Setting for All Attached Volumes

과제 1.4: 인스턴스 메모리와 디스크 캡처

인시던트 대응의 다음 단계는 인스턴스에 첨부된 영구 디스크의 스냅샷과 메모리 내용을 캡처하는 것입니다. 디스크 스냅샷 및 메모리 내용을 사용하여 새 인스턴스를 생성합니다. 이 인스턴스는 상세하게 포렌식 분석하여 인시던트의 근본 원인을 파악하는 데 사용됩니다.

인스턴스에 첨부된 볼륨의 스냅샷을 만들려면 인스턴스에 첨부된 볼륨을 식별해야 합니다.

  1. EC2 Management Console을 아직 사용하지 않은 경우 상단의 AWS Management Console 검색창에서

    Copied
    를 검색하여 선택합니다.

  2. 왼쪽 탐색 창에서 Elastic Block Store 섹션 아래에 있는 Volumes를 선택합니다.

  3. 손상된 인스턴스에 첨부된 볼륨을 식별하려면 Volumes 아래에서 오른쪽으로 스크롤하여 Attached Instances 열을 찾은 다음 compromise-ec2 인스턴스에 첨부된 볼륨을 찾습니다. 왼쪽으로 스크롤하여 손상된-ec2 볼륨에 대해 를 선택합니다.

  4. 페이지 오른쪽 상단에서 Actions 를 선택하고 Create snapshot을 선택합니다.

  5. Create snapshot 페이지에서 다음을 구성합니다.

  • Description에
    Copied
    을 입력합니다.
  • Tags에서 Add tag를 선택한 다음 Key 필드에
    Copied

과제 4: 인시던트 대응 작업을 처리하기 위한 자동화된 인시던트 대응 생성

과제 1과 3에서 수행한 단계는 EC2 인스턴스와 관련된 인시던트에 일반적입니다. 그러나 이러한 모든 과제를 수동으로 하는 것은 모범 사례가 아니므로 초기 대응을 자동화해야 합니다.

이 과제에서는 EC2 인스턴스와 관련된 모든 인시던트에 의해 트리거될 Amazon EventBridge 규칙을 생성하고 Lambda 함수를 호출하여 과제 1에서 수행한 단계를 자동 수행합니다.

이 과제에 사용되는 시나리오는 다음과 같습니다.

  • app-server에 대해 SSH 실패 시도가 시작됩니다. jump-server에 의해 시도가 시뮬레이션됩니다.
  • app-server가 SSH 인증 실패 로그를 생성하고 이 로그를 CloudWatch 로그 그룹으로 보냅니다.
  • CloudWatch 로그 그룹에는 SSH 인증 실패 로그와 일치하는 사전 구성된 구독 필터가 있으며 이를 사전 구성된 Lambda 함수(Log Enhancement Lambda)로 전송합니다.
  • 로그 보강 Lambda는 app-server 내부 IP 주소를 캡처하고 인스턴스 ID를 검색합니다. 그런 다음 이벤트를 인스턴스 ID가 포함된 Amazon EventBridge로 보냅니다.
  • EventBridge 규칙은 수신 이벤트와 일치하며 Automation Lambda가 자동화된 인시던트 대응 과제를 수행하도록 트리거합니다.

시나리오는 아래 다이어그램에 설명되어 있습니다.

이 과제에서 구성해야 하는 EventBridge 규칙을 제외하고 위의 모든 요구 사항은 실습 빌드에 미리 생성됩니다.

Automation Lambda 함수는 실습 빌드에서 미리 생성되며 이 함수의 예는 여기에서 확인할 수 있습니다.

과제 4.2: 다른 인스턴스에 대해 SSH 인증 실패를 시작하여 규칙을 테스트

이제 jump-server에서부터 실패한 SSH 시도를 app-server-2에서 의도적으로 시작하여 EventBridge 규칙을 테스트합니다. 이러한 시도를 시작하기 전에 app-server-2의 현재 설정을 확인하십시오.

  1. AWS 관리 콘솔 상단의 검색 창에서

    Copied
    를 검색하여 선택합니다.

  2. 왼쪽 탐색 창에서 Instances 섹션에서 Instances를 선택합니다.

  3. Instances에서 app-server-2 인스턴스를 찾아 이름 옆에 있는 를 선택합니다.

  4. 인스턴스 세부 정보의 Details 탭에서 아래로 스크롤하여 현재 Termination protection 상태를 확인합니다. 상태는 Disabled이어야 합니다.

  5. 인스턴스 세부 정보의 Security 탭에서 현재의 IAM Role 및 Security groups를 확인합니다. 다음 값으로 설정되어야 합니다.

  • IAM Role: app-server-role

  • Security groups: App-server-2-SG

  1. 인스턴스 세부 정보에서 Tags 탭을 선택합니다. 인스턴스에 Status:Quarantine 태그가 없다는 것을 확인합니다.

이는 실패한 SSH 시도를 시작하기 전 app-server-2의 기본 설정입니다. EventBridge 자동화가 올바르게 구성된 경우 자동 대응은 다음 작업을 수행해야 합니다.

  • 메타데이터를 캡처하여 IR-Bucket에 업로드합니다.
  • 인스턴스에서 종료 방지 기능을 활성화합니다.
  • 인스턴스 볼륨의 스냅샷을 생성합니다.
  • 인스턴스에서 Status:Quarantine 태그를 적용합니다.
  • 인스턴스에서 현재 IAM 역할을 제거합니다.
  • App-server-2-SG 보안 그룹을 제거하고 Quarantine-SG 보안 그룹을 적용하여 인스턴스를 격리합니다.

실패한 SSH 시도를 시작해 봅시다.

  1. jump-server 세션 관리자 브라우저 탭이 있으면 해당 탭으로 이동합니다. 없으면 이 지침 왼쪽에 있는 목록에서 SessionManagerJumpServerUrl 값을 복사하여 새 브라우저 탭에서 이 URL을 엽니다. 이렇게 하면 jump-server$ 셸 프롬프트와 함께 jump-server 터미널이 열립니다.

  2. 명령: 실패한 SSH 시도를 시작하려면 AppServer2DNSName을 이 지침의 왼쪽 목록에 있는 값으로 변경한 후 다음 명령을 실행합니다.

jump-server$

Copied 
sshpass -p "password" ssh -o StrictHostKeyChecking=no student@<REPLACE_WITH_YOUR_AppServer2DNSName>

  2. IR-Bucket을 확인하고 자동 대응의 일부로 메타데이터 파일이 버킷에 업로드되었는지 확인할 수도 있습니다.
